Глава 5. Frontend глазами аналитика

Почему браузер — не просто окно в базу данных, и как требования превращаются в интерфейс

💡 Метафора Backend — это кухня ресторана. Frontend — это зал, официанты, меню и тарелки. Можно приготовить идеальное блюдо (идеальный API, правильные данные), но если подача ужасна — тарелки грязные, официант грубый, меню нечитаемое — гость уйдёт недовольным. Хороший аналитик заботится не только о том, какие данные придут, но и как они будут выглядеть на экране пользователя.

Фронтенд — это то, что видит пользователь. И именно фронтенд определяет, вернётся ли пользователь завтра. Сколько бы ни был совершенен ваш бэкенд, если кнопка не нажимается, страница грузится 5 секунд, а на мобильном телефоне всё разъезжается — проект провалится.

В этой главе мы посмотрим на фронтенд не как разработчики, а как аналитики. Поймём, какие ограничения есть у браузера, как фронт общается с бэком, почему «просто показать данные из БД» — это сложная задача, и как правильно описывать требования к интерфейсу.

5.1. Что такое Frontend и чем он отличается от Backend

Фронтенд — это всё, что выполняется на устройстве пользователя: браузер (веб-приложение), мобильное приложение, десктопный клиент. Бэкенд — это код, который работает на сервере под вашим контролем.

Казалось бы, и там и там код. Но разница фундаментальна.

Главное отличие: доверие

Сформулируем жёстко, но точно:

❌ Заблуждение: «Фронтенд не доверяет бэкенду — мало ли, сервер вернёт не те данные».

💡 На самом деле: Бэкенд НЕ ДОВЕРЯЕТ фронтенду. Всегда. Никогда. Любые данные, пришедшие с клиента, могут быть подделаны, изменены, отправлены злоумышленником через Postman или cURL. Браузер — это среда, которую вы не контролируете.

Фундаментальный принцип веб-безопасности: входные данные от клиента — это враг. Проверка на фронте — для удобства пользователя. Проверка на бэке — для безопасности системы.

Сравнение Frontend и Backend

Критерий Frontend Backend
Где выполняется Браузер / устройство пользователя Сервер под вашим контролем
Языки JavaScript, TypeScript, HTML, CSS Java, Python, C#, Go, Rust, PHP, Kotlin...
Среда Неизвестная, неконтролируемая Известная, контролируемая
Безопасность Пользователь видит весь код (инспектор) Код скрыт, доступ только через API
Производительность Ограничена устройством пользователя Масштабируется (добавить серверов)
Обновление Сложно: нужно перезагрузить страницу или ждать кэш Просто: рестарт сервера (zero-downtime deploy)
Отладка Инструменты разработчика в браузере IDE, логи, мониторинг, профайлер
Доступ к данным Только через API (HTTP-запросы) Прямой доступ к БД, кэшу, файловой системе
Состояние Stateful (всегда есть текущее состояние) Stateless (каждый запрос — с нуля)
Юнит-тесты Сложнее (нужен браузер, DOM) Проще (чистая логика)

Что это значит для аналитика

Требование «при нажатии на кнопку сервер сохраняет данные» недостаточно. Аналитик должен понимать:

  • Как фронт узнает об успехе или ошибке?
  • Что показывать пользователю во время ожидания?
  • Что если пользователь нажал кнопку дважды?
  • Что если соединение прервалось?

📌 Золотое правило Каждый запрос с фронта на бэк — это история с неизвестным финалом. Фронт отправляет запрос и... может не получить ответа. Ответ может быть неверным. Ответ может прийти через 30 секунд. Хороший аналитик описывает все возможные исходы.

5.2. Как браузер получает страницу (очень кратко)

Аналитику не нужно знать, как работает V8-движок JavaScript. Но понимать жизненный цикл загрузки страницы — обязательно. Потому что производительность — это часть UX, а UX — это требования.

Путь от клика до отображения

Пользователь вводит URL в браузере │ ▼ 1. DNS-резолвинг: домен → IP-адрес сервера │ ▼ 2. HTTP-запрос (GET /index.html) │ ▼ 3. Сервер возвращает HTML-документ │ ▼ 4. Браузер парсит HTML, видит ссылки на CSS и JS │ ▼ 5. Загрузка CSS (блокирует рендеринг) │ ▼ 6. Загрузка JavaScript (блокирует парсинг) │ ▼ 7. Построение DOM-дерева + CSSOM → Render Tree │ ▼ 8. Первый рендер (First Paint) — пользователь видит страницу │ ▼ 9. Загрузка данных через API (XHR / Fetch) │ ▼ 10. Time to Interactive — страница полностью готова

Почему скорость — это деньги

🏦 Интернет-магазин: цена задержки Исследования Amazon и Google показывают:

  • 1 секунда задержки загрузки → падение конверсии на 7%.
  • 3 секунды → 53% мобильных пользователей покидают сайт.
  • 5 секунд → каждый второй клиент уходит к конкурентам.

Для интернет-магазина с оборотом 10 млн рублей в месяц потеря 7% конверсии — это 700 000 рублей в месяц. Уже не кажется, что «ну подождут пару секунд».

Что должен знать аналитик: метрики загрузки

Метрика Что измеряет Почему это важно
First Paint (FP) Первый пиксель на экране Пользователь видит, что страница загружается
First Contentful Paint (FCP) Первый контент (текст, картинка) Пользователь понимает, что сайт работает
Largest Contentful Paint (LCP) Загрузка основного контента Ключевой элемент страницы виден
Time to Interactive (TTI) Страница готова к взаимодействию Можно нажимать кнопки, заполнять формы

✅ В требованиях указывайте не только что показать, но и когда Если на странице должна отображаться таблица с 10 000 строк — не ждите, что она появится мгновенно. Укажите: «сперва показать skeleton screen, затем первые 50 строк, далее подгрузка по скроллу». Без этого требования разработчик загрузит все 10 000 строк сразу — и страница будет тормозить.

💡 Факт Google использует Core Web Vitals (LCP, FID, CLS) как фактор ранжирования. Медленный сайт = низкие позиции в поиске. Производительность — это не только UX, но и SEO.

5.3. API — интерфейс между Frontend и Backend

Это самый важный раздел для аналитика. Потому что API — это мост между требованиями к данным и их отображением. Если мост кривой — всё развалится.

Фронт НЕ ходит в базу данных

Это должен усвоить каждый аналитик. Браузер не может подключиться к PostgreSQL. У него нет драйвера БД, нет пароля, нет доступа к сети сервера. Всё, что умеет браузер — отправлять HTTP-запросы на сервер и получать ответы.

Браузер Сервер База данных │ │ │ │ HTTP GET /api/users │ │ │─────────────────────────▶│ │ │ │ SQL SELECT * FROM users │ │ │─────────────────────────▶│ │ │ │ │ │ ◀── rows ──────────────│ │ ◀── JSON-ответ ────────│ │ │ │ │ │ Отображение на экране │ │

Конфликт интересов: один endpoint vs много

⚠️ Типичный конфликт в команде

  • Фронтендер: «Дайте мне один endpoint /api/page/profile, который сразу вернёт и пользователя, и его заказы, и уведомления, и баланс. Зачем мне делать 5 запросов?»
  • Бэкендер: «Разделим на GET /users/me, GET /orders?userId=42, GET /notifications, GET /balance. Каждый endpoint делает одну простую вещь.»

У обоих аргументов есть логика:

Один «жирный» endpoint Много маленьких endpoint'ов
Плюсы Один запрос — все данные. Быстро, просто на фронте. Каждый endpoint переиспользуется. Лёгкое кэширование. Чистая архитектура.
Минусы Не переиспользуется. Если фронту нужны только заказы — придётся тащить всё. Множество запросов. Сложнее оркестрация на фронте.

Компромисс: BFF (Backend For Frontend)

BFF — это промежуточный слой, который собирает данные из разных микросервисов и отдаёт их в формате, удобном для конкретного клиента.

💡 BFF как переводчик Представьте, что у вас есть три гостя: англичанин, немец и японец. Вы можете говорить с каждым на его языке, а не заставлять всех учить английский. BFF — это переводчик между бэкендом и конкретным фронтендом.

Frontend (Web) Frontend (Mobile) Frontend (Tablet) │ │ │ │ /api/web/page │ /api/mob/page │ /api/tab/page ▼ ▼ ▼ ┌──────────────────────────────────────────────────────┐ │ BFF (Backend For Frontend) │ │ - Web BFF - Mobile BFF - Tablet BFF │ └──────────────────────────────────────────────────────┘ │ │ │ ▼ ▼ ▼ ┌──────────────────────────────────────────────────────┐ │ Микросервисы / Backend API │ │ Users │ Orders │ Payments │ Notifications │ Catalog │ └──────────────────────────────────────────────────────┘

🧠 Как думает Senior Каждый клиент (web, mobile, tablet, Smart TV) может требовать свой API. У мобильного приложения маленький экран и слабый интернет — ему нужно меньше данных, но быстрее. У десктопа — больше данных, сложнее UI. BFF позволяет оптимизировать ответ под каждого клиента, не ломая общий бэкенд. Как аналитик, вы должны явно указывать: для каких платформ предназначен эндпоинт и какие данные нужны каждой платформе.

5.4. Что видит пользователь ≠ что в БД

Одна из самых частых ошибок аналитика — описать требования к данным так, как они хранятся в базе, а не так, как они отображаются пользователю. Это разные вещи.

Пример: профиль пользователя

В базе данных На экране
user.id = 42 Не отображается (внутренний ID)
user.first_name = "Иван" «Иван Петров» (склеено)
user.last_name = "Петров"
user.role = "admin" Кнопка «Администрирование» (видна только админу)
user.created_at = 2023-05-12T14:30:00Z «12 мая 2023» (форматировано, локализовано)
user.avatar_url = "https://..." Картинка-аватар (не URL, а изображение)
user.is_active = true Не отображается напрямую

❌ Ошибка аналитика: «вывести все поля» Пользователю не нужен user.id, created_at в ISO-формате, is_active в виде булева значения. Ему нужен смысл данных, а не сырые значения. Указывайте:

  • какие поля показать;
  • в каком формате (дата, валюта, телефон);
  • какие поля скрыть.

Преобразования, которые происходят между БД и экраном

  • Форматирование: дата 2023-05-12 → «12 мая 2023», цена 1499.50 → «1 499,50 ₽»
  • Локализация: «Order placed» → «Заказ оформлен», «12/05/2023» vs «05/12/2023»
  • Агрегация: несколько полей → одно значение (например, first_name + last_name → полное имя)
  • Фильтрация: показать только активные заказы, скрыть удалённые
  • Трансформация: URL аватара → тег <img>; статус 1, 2, 3 → текст «Новый», «В работе», «Завершён»

✅ Best Practice: формат отображения — часть требований Не пишите «показать дату создания». Пишите: «показать дату создания в формате "12 мая 2023" (день месяц год, на русском языке)». Не пишите «показать цену». Пишите: «показать цену в формате "1 499,50 ₽" с разделителем тысяч и символом валюты». Это снимает десятки вопросов.

5.5. Состояние (State) — главная боль фронтенда

В бэкенде всё просто: пришёл запрос — обработал — отдал ответ — забыл. Каждый запрос независим. Бэкенд — stateless.

Фронтенд — наоборот: stateful. У него есть состояние, которое живёт между запросами: какие данные на странице, какой скролл, открыта ли модалка, выбран ли товар в корзине.

Что такое «состояние» на фронтенде

Тип состояния Пример Где хранится
Данные с бэка Список товаров, профиль, заказы Redux / MobX / React State
UI-состояние Открыт ли дропдаун, активная вкладка, модалка Локальный стейт компонента
Форма Заполненные поля, ошибки валидации Локальный стейт или библиотека форм
Скролл Позиция скролла в ленте новостей DOM / JavaScript
Кэш Ранее загруженные данные (чтобы не грузить снова) LocalStorage / Cache API / память

Проблема: рассинхронизация состояний

Состояние на фронте и состояние на бэке могут разойтись. Это главная головная боль фронтенд-разработчиков.

🏦 Пример: корзина интернет-магазина Пользователь добавил товар в корзину. На фронте появилась плашка «Товар добавлен (3 шт.)». Но бэкенд вернул ошибку: «товара нет в наличии». Что делать?

  • Вариант A: откатить состояние корзины (вернуть как было) — пользователь удивится, почему товар исчез.
  • Вариант B: показать ошибку, но оставить товар (выделив его красным) — пользователь увидит, что что-то пошло не так.
  • Вариант C: заблокировать кнопку «добавить» до ответа сервера — пользователь не сможет ничего делать.

Аналитик должен выбрать один из вариантов и описать его в требованиях. Если не описать — разработчик выберет самый простой (а не самый правильный).

❌ Ошибка: «почему при обновлении страницы данные пропали?» Потому что состояние фронта — временное. Если вы не сказали аналитику «сохранить корзину в LocalStorage» или «восстановить данные из кэша» — при перезагрузке страницы всё сбросится. Описывайте, что должно сохраняться между сессиями.

🧠 Как думает Senior Всегда думайте о том, как фронт узнает об изменениях на бэке. Если менеджер изменил статус заказа в админке — как об этом узнает пользователь? Через WebSocket? Через polling каждые 5 секунд? Через pull-to-refresh? Не оставляйте это на усмотрение разработчика. Опишите механизм синхронизации.

5.6. Загрузка и индикаторы (UX = часть требований)

Загрузка — это не техническая деталь, а часть пользовательского опыта. Аналитик должен описывать, что видит пользователь во время каждого из возможных состояний компонента.

Три состояния каждого компонента

Состояние Когда Что показать Пример
Загрузка Данные запрошены, но ещё не пришли Skeleton screen или спиннер Серые прямоугольники на месте будущего контента
Успех Данные получены, всё ок Нормальный интерфейс Таблица с товарами, профиль пользователя
Ошибка Запрос завершился ошибкой Сообщение об ошибке с кнопкой повтора «Не удалось загрузить данные. Попробовать снова»

✅ Best Practice: описывайте все три состояния В требованиях к каждому экрану указывайте:

  • что показывается, пока данные грузятся (скелет / спиннер / анимированный лоадер);
  • что показывается, когда данные загружены успешно;
  • что показывается, если произошла ошибка (текст, иконка, кнопка повтора).

Если не указать состояние загрузки — разработчик покажет пустой экран, пока данные не придут. Если не указать состояние ошибки — пользователь увидит бесконечный спиннер.

Когда что показывать

Тип индикатора Когда использовать Пример
Skeleton screen Загрузка основного контента страницы Лента новостей, профиль, список заказов
Спиннер (локальный) Загрузка конкретного блока / действия Кнопка «Сохранить» со спиннером внутри
Прогресс-бар Длительная операция с известным прогрессом Загрузка файла, импорт 1000 строк
Fullscreen спиннер Критическая загрузка (блокирующая) Оплата, редирект после логина

⚠️ Чего нельзя делать «Просто покажи ошибку» — этого недостаточно. Ошибка сети (нет интернета), ошибка сервера (500), ошибка авторизации (401), ошибка валидации (400), таймаут — всё это разные ситуации. Для каждой нужно описать:

  • какой текст показать;
  • какую иконку/цвет;
  • можно ли повторить действие;
  • куда ведёт кнопка повтора.

🏦 Пример: ошибка оплаты Пользователь оплачивает заказ. Сервер вернул 500. Что делает фронт?

  • Плохо: показать «Ошибка 500. Попробуйте позже». Пользователь в панике: деньги списались или нет?
  • Хорошо: показать «Не удалось обработать платёж. Пожалуйста, проверьте статус заказа в разделе "Мои заказы". Если деньги списаны, они вернутся в течение 1-3 дней.»

Разница в том, что второй текст управляет ожиданиями пользователя.

5.7. Асинхронность — пользователь не ждёт

Пользователь — существо нетерпеливое. Он не будет ждать ответа сервера, чтобы нажать кнопку ещё раз. Он не будет сидеть и смотреть на спиннер. Он будет кликать, тапать, тыкать.

Фронтенд должен быть спроектирован с учётом асинхронности — то есть того факта, что между действием пользователя и ответом сервера проходит время, и пользователь может сделать ещё много всего за это время.

Проблема: множественные нажатия

🏦 Пример: двойной платёж Пользователь нажимает «Оплатить». Сервер не отвечает 3 секунды. Пользователь нажимает ещё раз. И ещё. В результате — три списания с карты. Фронтенд должен блокировать кнопку после первого нажатия, пока не придёт ответ сервера.

❌ Не блокировать кнопку отправки — допустить двойной платёж Это одна из самых дорогих ошибок в e-commerce. Требование: «кнопка отправки должна быть заблокирована после нажатия до получения ответа сервера» — должно быть в каждом ТЗ, где есть отправка данных.

Защитные механизмы

Техника Как работает Когда использовать
Disabled-состояние кнопки Кнопка становится неактивной после первого нажатия Оплата, отправка формы, любое «тяжелое» действие
Debounce Действие выполняется только спустя X мс после последнего вызова Поиск при вводе текста (не отправлять запрос на каждый символ)
Throttle Действие выполняется не чаще 1 раза в X мс Обработка скролла, ресайза окна
Idempotency Key На бэкенде: повторный запрос с тем же ключом не создаёт новый ресурс Создание заказа, платежа (защита на уровне API)

Оптимистичный интерфейс

Оптимистичный интерфейс — это когда фронт предполагает, что запрос выполнится успешно, и обновляет интерфейс сразу, не дожидаясь ответа сервера. Если сервер вернул ошибку — фронт «откатывает» изменения.

🏦 Пример: лайк в соцсети Вы нажимаете сердечко — оно сразу загорается. Вы даже не замечаете, что в этот момент ушёл HTTP-запрос. Если сервер упадёт, сердечко погаснет через секунду, но пользователь уже получил мгновенный отклик. Это и есть оптимистичный интерфейс.

⚠️ Когда оптимизм опасен Оптимистичный интерфейс хорош для «лайков», комментариев, добавления товара в корзину. НО он опасен для финансовых операций. Никогда не показывайте «оплата прошла» до ответа платёжного шлюза. Для операций с деньгами — только пессимистичный подход: ждать ответа сервера.

5.8. Формы и валидация на фронтенде

Формы — это основной способ взаимодействия пользователя с системой. Регистрация, оформление заказа, создание задачи — всё это формы. И у форм есть свои правила.

Два уровня валидации

Уровень Цель Когда срабатывает Пример
Frontend UX — мгновенная обратная связь При вводе или при отправке «Email должен содержать @», «Поле обязательно»
Backend Безопасность — защита от подделки запроса При получении запроса «Email уже занят», «Пароль слишком короткий»

💡 Запомните формулу Фронт валидирует для удобства. Бэк валидирует для безопасности. Фронт можно обойти (отключить JavaScript, отправить запрос через Postman). Бэк — последняя линия обороны.

Типичные фронтенд-валидации

Правило Пример Сообщение пользователю
Required Имя обязательно «Пожалуйста, заполните это поле»
Email user@example.com «Введите корректный email-адрес»
Min/Max length Пароль от 8 до 64 символов «Пароль должен содержать от 8 до 64 символов»
Regex pattern Телефон +7 (999) 123-45-67 «Введите номер в формате +7 (999) 123-45-67»
Range Количество от 1 до 99 «Количество должно быть от 1 до 99»
Confirm Пароль == Подтверждение пароля «Пароли не совпадают»

💡 Совет Если не знаете, когда показывать ошибку — показывайте на onBlur (при потере фокуса). Это золотая середина: пользователь уже заполнил поле, но ещё не отправил форму. Он видит ошибку сразу, но она не «дёргает» его на каждый символ.

✅ Best Practice: когда показывать ошибки

  • После отправки формы — если ошибок много, показать все сразу. Подсветить каждое поле.
  • После потери фокуса (onBlur) — для отдельных полей. Пользователь заполнил email, перешёл к следующему полю — и сразу увидел ошибку.
  • В реальном времени (onChange) — для полей с подсказками (сила пароля, количество символов).

Указывайте момент валидации в требованиях. «Неверный формат email» может появиться сразу при вводе или только после нажатия «Отправить» — это разные UX.

🧠 Как думает Senior Хорошее сообщение об ошибке = что не так + как исправить. Плохо: «Ошибка в поле email». Хорошо: «Введите email в формате user@example.com». Плохо: «Некорректный пароль». Хорошо: «Пароль должен содержать хотя бы одну заглавную букву и цифру». Аналитик, который пишет такие сообщения в требованиях, — на вес золота.

5.9. Mobile First и Адаптивность

В 2024 году более 60% трафика в мире приходится на мобильные устройства. Если ваш интерфейс не работает на телефоне — вы теряете больше половины пользователей.

Что значит Mobile First для аналитика

Mobile First — подход, при котором дизайн начинается с мобильного экрана, а затем расширяется до десктопа. Не наоборот.

💡 Логика простая Втиснуть десктопный интерфейс в мобильный экран невозможно. Но расширить мобильный интерфейс до десктопа — легко. Начинайте с малого и добавляйте, а не начинайте с большого и вырезайте.

Responsive vs Adaptive

Подход Как работает Когда использовать
Responsive Один и тот же интерфейс «течёт» под размер экрана (резиновая вёрстка) Контентные сайты, блоги, новости
Adaptive Несколько фиксированных макетов для разных разрешений (breakpoints) Сложные интерфейсы, панели управления, дашборды

Ограничения мобильных устройств

  • Размер экрана: 320–430px в ширину. Таблицы, много колонок, мелкие кнопки не работают.
  • Скорость интернета: мобильный интернет может быть в 10 раз медленнее Wi-Fi. Не грузите 5 МБ изображений.
  • Трафик: пользователи могут платить за мегабайты. Предзагрузка видео без спроса — плохая идея.
  • Тач-взаимодействие: нет наведения (hover). Нет правой кнопки. Есть свайпы, долгие нажатия.

🏦 Пример: мобильное приложение банка У банка одни и те же API для веба и мобильного приложения. Но UI — совершенно разный. На десктопе — таблица с 20 столбцами и графиками. В мобильном приложении — карточки с ключевыми метриками и стрелочка «подробнее». Аналитик должен описать оба интерфейса, а не копировать десктопный на мобилку.

⚠️ Типичная ошибка: «на мобилке тоже будет работать» Нет, не будет. Если вы описали интерфейс только для десктопа — мобильная версия будет сделана «как получится». Описывайте адаптацию:

  • какие элементы скрывать на мобилке;
  • как перестраивать навигацию (гамбургер-меню);
  • как трансформировать таблицы (карточки вместо строк);
  • как размещать кнопки (удобно для большого пальца).

5.10. Кэширование на фронтенде

Фронтенд может (и должен!) кэшировать данные, чтобы не дёргать сервер без необходимости. Но кэш на фронте — это палка о двух концах.

Типы кэша на фронте

Тип Где хранится Жизнь Объём Когда использовать
LocalStorage Браузер, диск Постоянно (до очистки) 5–10 МБ Настройки, токены, состояние корзины
SessionStorage Браузер, диск До закрытия вкладки 5–10 МБ Временные данные формы, черновики
Cache API Браузер, диск До инвалидации До ~50 МБ Статические ресурсы, API-ответы (Service Worker)
In-memory (Redux, React state) Оперативная память До перезагрузки страницы Зависит от ОЗУ Текущие данные приложения
HTTP-кэш браузера Браузер, диск По HTTP-заголовкам (Cache-Control, ETag) Настраивается Изображения, CSS, JS-файлы
Service Workers Браузер, фон Программируемое Настраивается PWA, офлайн-режим, продвинутое кэширование

Чем отличается кэш на фронте от кэша на бэке

  • Бэкенд-кэш (Redis, Memcached) — централизованный, живёт на сервере, им управляют разработчики. Можно очистить в любой момент.
  • Фронтенд-кэш — распределённый, живёт на устройствах пользователей. Вы не можете его очистить удалённо. Вы можете только попросить браузер обновить данные через HTTP-заголовки.

Проблема: устаревшие данные

❌ Ошибка: пользователь видит старые данные Вы обновили список товаров на сервере, но пользователь всё ещё видит старую версию, потому что браузер закэшировал ответ API. Как заставить браузер обновить данные?

Методы инвалидации кэша:

  • Cache-Control: no-cache — проверять свежесть при каждом запросе.
  • ETag / Last-Modified — сервер проверяет, изменились ли данные, и возвращает 304 Not Modified, если нет.
  • Версионирование URL — /api/v2/products вместо /api/v1/products.
  • Query-параметр версии — /api/products?version=2.
  • WebSocket / Server-Sent Events — сервер сам «толкает» обновление на фронт.

📌 Для аналитика Если данные могут меняться без участия пользователя (например, статус заказа меняет оператор), опишите, как фронт узнаёт об этом. «Пользователь должен видеть актуальный статус» — это не требование. «Фронт должен проверять статус заказа каждые 10 секунд (polling)» — это требование.

5.11. Что аналитик должен знать про фронтенд (практические советы)

Соберём всё, что мы узнали, в практические рекомендации для аналитика.

10 правил для аналитика при работе с фронтендом

  1. Не просить «вывести все». Список из 50 полей убивает производительность и UX. Только то, что нужно пользователю на этом экране.
  2. Указывать состояние компонента. Загрузка, пусто, ошибка, успех — опишите все четыре сценария.
  3. Описывать поведение при сетевой ошибке. Что видит пользователь, если интернет пропал? Что видит, если сервер ответил 500?
  4. Указывать формат отображения. Не «дата», а «12 мая 2023». Не «цена», а «1 499,50 ₽». Не «статус», а зелёный кружок «Активен».
  5. Помнить про мобильные устройства. Ваш интерфейс будет работать на экране шириной 360 пикселей с нестабильным интернетом.
  6. Указывать, что НЕ показывать. Пароли, хеши, внутренние ID, служебные флаги — явно перечислите, что должно быть скрыто.
  7. Указывать, что показывать, когда ничего не показывается. Пустой список ≠ ошибка. Напишите: «Если заказов нет, показать заглушку: "У вас пока нет заказов" с кнопкой "Начать покупки"».
  8. Описывать поведение кнопок при отправке. Блокируется ли кнопка? Показывается ли спиннер? Что при ошибке?
  9. Учитывать кэш. Если данные обновляются редко, укажите, можно ли их кэшировать и на сколько.
  10. Проверять крайние случаи. Пользователь ввёл 1000 символов вместо 100. Пользователь загрузил файл 100 МБ вместо 5 МБ. Что делает интерфейс?

🧠 Резюме для Senior Analyst Хороший аналитик не пишет «показать данные». Он пишет: «Показать заголовок страницы, пока данные грузятся — skeleton screen (три серых прямоугольника). После загрузки — карточку товара с изображением, названием, ценой (формат "1 499,50 ₽") и кнопкой "В корзину". При ошибке — баннер "Не удалось загрузить товар" с кнопкой "Повторить". Если корзина пуста — заглушку "Корзина пуста" с кнопкой "Перейти к покупкам". Кнопку "В корзину" блокировать на 1 секунду после нажатия (debounce). На мобильных устройствах изображение показывать в полную ширину экрана, таблицу характеристик — скрыть под спойлер "Все характеристики ▼".»

Каждый пункт здесь — это решение, которое разработчику не придётся принимать за вас.

Итоги главы

📌 Главная мысль Фронтенд — это не «просто отображение данных из БД». Это полноценная среда со своими ограничениями, состоянием, асинхронностью и правилами безопасности. Чем лучше аналитик понимает фронтенд, тем точнее его требования и тем меньше вопросов к нему будет у разработчиков.

Что должен понять аналитик

  1. Фронтенд не доверяет бэкенду? Нет, бэкенд не доверяет фронтенду. Любые данные от клиента могут быть подделаны.
  2. Скорость — часть UX. 1 секунда задержки = -7% конверсии. Производительность должна быть в требованиях.
  3. Фронт не ходит в БД. Только через API. А API — это предмет переговоров между фронтом и бэком.
  4. Что видит пользователь ≠ что в БД. Форматирование, локализация, агрегация — опишите это в требованиях.
  5. Состояние фронта — временное. Оно может рассинхронизироваться с бэком. Опишите, что делать при рассинхронизации.
  6. Три состояния каждого компонента: загрузка, успех, ошибка. Опишите все три.
  7. Пользователь не ждёт. Debounce, блокировка кнопок, защита от двойного нажатия — это требования, а не «технические детали».
  8. Валидация на фронте — для UX. Настоящая защита — на бэке.
  9. Mobile First — не мода, а необходимость. 60%+ трафика — мобильные устройства.
  10. Кэш на фронте неуправляем. Вы не можете очистить его удалённо — только подсказать браузеру.

Чек-лист: что проверить в требованиях касательно фронтенда

  • Указаны все состояния компонента (загрузка, пусто, ошибка, успех)?
  • Описано поведение при сетевой ошибке и ошибке сервера?
  • Указан формат отображения данных (дата, валюта, телефон, статус)?
  • Описан формат ошибок (что писать пользователю, какой цвет, иконка)?
  • Указано поведение кнопок при отправке (блокировка, спиннер)?
  • Описана адаптация под мобильные устройства?
  • Указано, какие данные кэшировать и как часто обновлять?
  • Описаны крайние случаи (длинный ввод, большой файл, пустой список)?
  • Указаны поля, которые НЕ нужно показывать (security)?
  • Описана синхронизация: как фронт узнает об изменениях на бэке?

🧠 Резюме для Senior Analyst Фронтенд — это мост между данными и пользователем. Хороший аналитик не просто описывает, что показать, а продумывает весь путь данных: от нажатия кнопки до появления информации на экране, включая все промежуточные состояния, ошибки и задержки. Когда вы научитесь мыслить категориями состояния, загрузки и асинхронности — вы перестанете быть «писцом требований» и станете архитектором пользовательского опыта.
