Глава 2. REST API без магии

Как перестать бояться эндпоинтов и начать проектировать API, которые не ломаются

Введение

Представьте, что вы заходите в большой отель. Вы подходите к стойке регистрации и говорите: «Я хочу заселиться в номер 404, бронирование на имя Иванова». Портье проверяет систему, находит запись, выдаёт ключ-карту — и вы идёте в номер. Вы не видите, что происходит за стойкой: какая база данных хранит брони, как система проверки оплаты связана с модулем расчёта налогов, как служба горничных узнаёт, что номер готов. Для вас есть только стойка регистрации — единая точка входа.

REST API — это и есть такая стойка регистрации. Клиент (фронтенд, мобильное приложение, сторонний сервис) приходит с запросом, сервер обрабатывает его и возвращает результат. За стойкой может быть вся сложность мира, но для клиента есть только чётко определённый интерфейс.

Цель этой главы — не просто перечислить методы HTTP или правила именования эндпоинтов. Вы должны научиться чувствовать REST API: понимать, почему он устроен именно так, какие решения принимает разработчик и — главное — как самому спроектировать API так, чтобы его не хотелось переписать через месяц.

Эта глава написана для аналитика, которому предстоит:

  • описывать контракты API в документации;
  • задавать правильные вопросы разработчикам;
  • принимать решения о дизайне новых эндпоинтов;
  • понимать, почему «вдруг всё сломалось» после изменения API.

2.1. Что такое REST API на самом деле

Аналогия с библиотекой

Представьте библиотеку. В библиотеке есть книги — это ресурсы. Есть каталог, по которому можно найти книгу, — это эндпоинты. И есть библиотекарь, который достаёт книгу, проверяет ваш читательский билет и фиксирует выдачу — это сервер.

Вы не кричите библиотекарю: «Эй, найди мне книгу!» Вы обращаетесь к каталогу: «Дайте мне книгу с ISBN 978-5-699-12345-6». Библиотекарь идёт, проверяет, есть ли книга, и возвращает вам её (или сообщает, что книги нет на месте).

REST (Representational State Transfer) — это архитектурный стиль, который описывает правила такого взаимодействия. Ключевая идея: в центре внимания — ресурсы, а не действия.

Факт. Термин REST предложил Рой Филдинг в своей докторской диссертации в 2000 году. Он же был одним из создателей протокола HTTP/1.1. Так что REST и HTTP — не случайные соседи, они рождались вместе.

Ресурс — центральное понятие

В REST всё строится вокруг ресурса. Ресурс — это некая сущность, к которой можно обратиться по уникальному адресу (URI). Что может быть ресурсом:

  • Заказ — /api/orders/12345
  • Пользователь — /api/users/42
  • Товар — /api/products/sku-77812
  • Платёж — /api/payments/txn-abc-001
  • Корзина покупателя — /api/carts/user-42
  • Отчёт по продажам — /api/reports/daily-sales

Важно. Ресурс — это существительное. Не «создать заказ» (глагол), а «заказы» (существительное). REST — это архитектура существительных, а не глаголов.

Почему REST — это не про глаголы?

Когда вы проектируете API, вам хочется назвать эндпоинты логично: /createOrder, /getUserList, /deleteItem. Это кажется естественным — мы же описываем действия. Но проблема в том, что действий десятки, а глаголов в языке — сотни. Каждый разработчик назовёт их по-своему: /getOrders, /fetchOrders, /retrieveOrders, /listOrders. Хаос.

REST предлагает другой подход: ресурсов ограниченное количество, а действия над ними описываются HTTP-методами. Вместо /createOrder — POST /api/orders. Вместо /deleteOrder — DELETE /api/orders/{id}.

HTTP-методы + Ресурсы = REST API Ресурс: /api/orders GET /api/orders → Получить список заказов POST /api/orders → Создать новый заказ GET /api/orders/1 → Получить заказ №1 PUT /api/orders/1 → Полностью заменить заказ №1 PATCH /api/orders/1 → Частично обновить заказ №1 DELETE /api/orders/1 → Удалить заказ №1

Запомните. Если в URL эндпоинта есть глагол — что-то пошло не так. Правильный URL состоит из существительных (ресурсов) и их идентификаторов.

2.2. HTTP-методы — не просто GET и POST

Большинство людей, далёких от бэкенда, знают два метода: GET — «получить данные», POST — «отправить данные». В реальности их пять основных, и каждый несёт строго определённый смысл. Аналитик обязан их различать.

Метод Действие Идемпотентность Безопасность Тело запроса Пример
GET Получить ресурс(ы) Да Да Нет GET /api/users/1
POST Создать ресурс Нет Нет Да POST /api/users
PUT Полностью заменить Да Нет Да PUT /api/users/1
PATCH Частично обновить Нет* Нет Да PATCH /api/users/1
DELETE Удалить ресурс Да Нет Нет** DELETE /api/users/1

* PATCH может быть идемпотентным, если операция замены задана абсолютно, но спецификация не гарантирует этого.
** Тело у DELETE технически допустимо, но лучшие практики запрещают его использовать.

Разбираем каждый метод

GET — получить (безопасно и идемпотентно)

GET используется только для чтения данных. Он безопасен — то есть не меняет состояние сервера. Его можно вызывать сколько угодно раз — результат будет одинаковым (идемпотентность). Именно поэтому GET-запросы кешируются, индексируются поисковиками и не требуют подтверждения.

Правило: если эндпоинт что-то меняет (списывает деньги, создаёт запись, отправляет письмо) — это не GET.

POST — создать (не идемпотентно)

POST — единственный метод, который не является идемпотентным по определению. Отправляя один и тот же POST-запрос дважды, вы рискуете получить два разных результата (например, два заказа).

Важно для аналитика: если вы проектируете платёжный эндпоинт, POST без защиты от дубликатов — это путь к двойным списаниям. Об этом мы поговорим в разделе 2.7.

PUT — полностью заменить (идемпотентно)

PUT заменяет весь ресурс новой версией. Если вы отправите один и тот же PUT-запрос дважды — результат будет одинаковым. PUT идемпотентен, потому что он закрепляет конкретное состояние ресурса.

Антипаттерн. Использовать PUT для частичного обновления (PUT /api/users/1 с телом {"name": "Новое имя"}, ожидая, что остальные поля не изменятся). Это ломает идемпотентность: повторный PUT может неожиданно перезаписать данные, которых не было в первом запросе. Для частичного обновления есть PATCH.

PATCH — частично обновить

PATCH появился в HTTP позже остальных, но быстро стал незаменимым. Он обновляет только те поля, которые переданы в теле запроса. Остальные поля остаются без изменений.

DELETE — удалить

DELETE удаляет ресурс. Идемпотентен: повторный DELETE вернёт 404 (ресурса уже нет), но это нормально — сервер не сломается.

Типичная ошибка. DELETE с телом запроса. DELETE не должен иметь тела. Если вам нужно передать причину удаления или другие параметры — используйте заголовки или query-параметры. Например: DELETE /api/orders/1?reason=duplicate (хотя лучше POST /api/orders/1/cancel).

Почему это важно для аналитика

Выбирая метод, вы задаёте семантику эндпоинта. Клиент (фронтенд, другой сервис) полагается на эту семантику. Если вы повесите удаление на GET — кеширующий прокси решит, что это безопасный запрос, и закеширует результат. Если вы используете POST для получения данных — потеряете кеширование и идемпотентность. Метод — это контракт.

Реальный пример: API банковского перевода.

Предположим, мы проектируем эндпоинт для перевода средств между счетами.

  • Неправильно: GET /api/transfer?from=123&to=456&amount=1000 — GET безопасный, его может выполнить поисковый робот или браузер при prefetch'е. Деньги уйдут просто от того, что кто-то перешёл по ссылке.
  • Неправильно: POST /api/transfer — если клиент потеряет соединение и повторит запрос, перевод выполнится дважды.
  • Правильно: POST /api/transfers с заголовком Idempotency-Key: uuid. Сервер проверяет ключ: если такой перевод уже был — возвращает сохранённый результат, не выполняя повторного списания.

2.3. Endpoint — это существительное, не глагол

Это настолько важное правило, что оно стоит отдельного раздела. URL в REST API — это путь к ресурсу, а не описание действия.

Правильно vs Неправильно

Неправильно (глагол) Правильно (существительное) Метод
/api/getOrders /api/orders GET
/api/createOrder /api/orders POST
/api/updateOrder /api/orders/1 PATCH
/api/deleteOrder /api/orders/1 DELETE
/api/getUserOrders?id=1 /api/users/1/orders GET
/api/changePassword /api/users/1/password PATCH

Аналогия. Представьте шкаф с папками. Каждая папка подписана: «Заказы 2024», «Клиенты», «Счета». Вы не пишете на папке «Положить сюда новый заказ» или «Взять отсюда заказ». Вы просто подходите к папке «Заказы» и делаете то, что нужно: кладёте новый бланк (POST), вынимаете старый (GET), заменяете содержимое (PUT) или выбрасываете папку (DELETE). URL — это имя папки, а не инструкция.

Почему это важно

  1. Единообразие. Все эндпоинты следуют одному шаблону. Не нужно гадать, как называется эндпоинт для обновления — метод меняется, а путь остаётся тем же.
  2. Предсказуемость. Зная структуру API, клиент может угадать эндпоинт для нового ресурса. Если есть /api/users и /api/orders, то будет и /api/products.
  3. Масштабирование. Добавление нового метода (например, PATCH) не ломает существующие эндпоинты. Путь не меняется — меняется только HTTP-метод.
  4. Инструментальная поддержка. Swagger, OpenAPI, клиентские генераторы — все они рассчитывают на существительные.

RESTful naming conventions

Несколько правил именования, которые помогут сделать API читаемым:

  • Множественное число для коллекций: /api/users, /api/orders (а не /api/user).
  • Вложенность для связанных ресурсов: /api/users/1/orders/5/items.
  • kebab-case для составных имён: /api/shipping-addresses (а не /api/shippingAddresses или /api/shipping_addresses).
  • Без глаголов: никаких /api/getAllUsers.
  • Без точек и расширений: /api/users/1, а не /api/users/1.json.
  • Параметры — в query, не в path: /api/orders?status=paid, а не /api/orders/status/paid.

Лучшая практика. Если вам кажется, что без глагола не обойтись — подумайте, не создаёте ли вы новый ресурс. Например, POST /api/orders/1/cancel можно заменить на POST /api/orders/1/cancellations, где «отмена» — это самостоятельный ресурс.

2.4. HTTP Status Codes — язык без слов

Код ответа HTTP — это первое, что видит клиент после запроса. Он говорит: «Всё хорошо» или «Что-то пошло не так». Аналитик должен знать коды не хуже разработчика, потому что именно аналитик описывает в документации, что и когда возвращается.

Категории статусов

Диапазон Категория Смысл
1xx Informational Запрос получен, продолжается обработка
2xx Success Запрос успешно обработан
3xx Redirection Клиенту нужно выполнить дополнительное действие
4xx Client Error Ошибка на стороне клиента
5xx Server Error Ошибка на стороне сервера

Самые важные коды (которые нужно знать наизусть)

Код Название Когда использовать
200 OK Запрос выполнен успешно. Для GET — тело содержит запрошенные данные. Для POST — тело содержит результат операции.
201 Created Ресурс успешно создан (POST). В ответе — созданный ресурс и заголовок Location с URL нового ресурса.
204 No Content Запрос выполнен успешно, но тело ответа пусто. Используется для DELETE, иногда для PUT/PATCH, когда обновление не требует возврата данных.
301 Moved Permanently Ресурс перемещён навсегда. Клиент должен обновить URL.
304 Not Modified Клиент может использовать кешированную версию.
400 Bad Request Сервер не может обработать запрос из-за ошибки клиента: невалидный JSON, неверный формат, пропущенные поля.
401 Unauthorized Клиент не предоставил учётные данные (или они неверны). Нужен токен / логин-пароль.
403 Forbidden Клиент аутентифицирован, но не имеет прав на выполнение операции.
404 Not Found Ресурс не найден. Может означать как отсутствие конкретной записи, так и неверный URL.
409 Conflict Конфликт с текущим состоянием ресурса. Например, попытка создать дубликат или удалить объект, на который есть ссылки.
422 Unprocessable Entity Запрос синтаксически верен (JSON валидный), но семантически некорректен: отрицательная цена, дата рождения в будущем.
429 Too Many Requests Превышен лимит запросов (rate limiting).
500 Internal Server Error Внутренняя ошибка сервера. Клиент не виноват, сервер сломался.
502 Bad Gateway Сервер (обычно прокси) получил некорректный ответ от вышестоящего сервера.
503 Service Unavailable Сервер временно недоступен (перегрузка, техническое обслуживание).

Как думает Senior. Аналитик, который различает 401 и 403, сразу вызывает уважение разработчиков. 401 — «кто ты?» (нужна аутентификация). 403 — «ты кто, но тебе нельзя» (не хватает прав). Путать их — значит путать идентификацию с авторизацией. Разные сценарии, разная логика обработки на клиенте. При 401 клиент показывает форму логина. При 403 — сообщение «доступ запрещён».

Антипаттерн: 200 на ошибку. Некоторые разработчики возвращают 200 OK всегда, а результат помещают в тело: {"success": false, "error": "Недостаточно средств"}. Проблема в том, что HTTP-статус теряет смысл. Кеширующие прокси не узнают об ошибке. Клиент должен разбирать тело ответа, чтобы понять, что пошло не так. Это путь к хаосу. Если произошла ошибка — верните соответствующий 4xx или 5xx код.

2.5. URL-дизайн: структура и иерархия

Хороший URL читается как путь к файлу. Плохой URL — как строка с секретным кодом.

Ресурсы и подресурсы

Связанные ресурсы вкладываются друг в друга. Это отражает их иерархию в предметной области:

/api/users/{userId}/orders/{orderId}/items/{itemId}

Читается как: «у пользователя №42 есть заказ №5, а в нём — позиция №1». Каждый уровень сужает контекст.

Глубина вложенности. Не увлекайтесь. Три уровня — предел. Если нужно глубже — используйте query-параметры или плоскую структуру.

Лучшая практика. Плоская структура с фильтрацией часто предпочтительнее глубокой вложенности. Вместо /api/users/1/orders/5/items/3/attachments/7 — /api/attachments?orderItemId=3.

Path vs Query: что куда

Идёт в URL (path) Идёт в query-параметры
Идентификатор ресурса (userId, orderId) Фильтры (status, dateFrom, category)
Имя ресурса (users, orders, products) Пагинация (page, size, offset)
Действие в виде подресурса (cancel, activate) Сортировка (sort, order)
Версия API (/api/v1/) Поиск (q, search, query)
Локализация (/en/, /ru/) Формат ответа (?format=json) — хотя лучше через заголовки Accept

Версионирование

API меняется, и клиенты должны к этому адаптироваться. Версионирование через URL — самый простой и понятный способ:

/api/v1/orders /api/v2/orders

Когда вы выпускаете v2, v1 продолжает работать для старых клиентов. Да, это значит, что вы поддерживаете две версии. Но это лучше, чем ломать работающие интеграции.

Альтернативы: версионирование через заголовки (Accept: application/vnd.api+json;version=2) или query-параметры (?version=2). Но URL-версия — самый прозрачный вариант.

Примеры правильных и неправильных URL

Неправильно Правильно
/api/getAllOrders?userId=42 /api/users/42/orders
/api/orders?delete=true&id=5 DELETE /api/orders/5
/api/order/create POST /api/orders
/api/UpdateStatusOfOrder PATCH /api/orders/123/status
/api/users/listOrders?userId=1 /api/users/1/orders
/api/products/sortByPrice /api/products?sort=price,asc
/api/items/byCategory?cat=books /api/categories/books/items

Реальный пример: API интернет-магазина.

Спроектируем фрагмент API для интернет-магазина:

Описание Метод и URL
Список товаров с фильтром по категории GET /api/v1/products?categoryId=5&page=1&size=20
Карточка товара GET /api/v1/products/sku-001
Добавление товара в корзину POST /api/v1/cart/items с телом: {productId, quantity}

Получение статуса заказа GET /api/v1/orders/ORD-12345
Оплата заказа POST /api/v1/orders/ORD-12345/payments с Idempotency-Key
Отмена заказа POST /api/v1/orders/ORD-12345/cancellations
История заказов пользователя GET /api/v1/users/42/orders?sort=date,desc
Получение отзыва на товар GET /api/v1/products/sku-001/reviews

2.6. Контракты API — почему их нельзя менять

Аналогия. API — это электрическая розетка. Форма розетки не менялась десятилетиями, хотя за стенами электростанции могут модернизироваться хоть каждый день. Клиент (вилка) должен подходить к розетке (API) без напильника.

Контракт API — это соглашение между клиентом и сервером. Клиент обещает отправлять запросы определённого формата. Сервер обещает возвращать ответы определённого формата. Изменение контракта без предупреждения — это нарушение доверия.

Обратная совместимость

Изменение считается обратно совместимым (не ломающим), если старые клиенты продолжают работать без изменений. Что можно делать:

  • Добавлять новые поля в ответ (клиенты, которые не читают эти поля, не сломаются).
  • Добавлять новые эндпоинты.
  • Добавлять новые опциональные параметры запроса.
  • Расширять диапазон допустимых значений (например, увеличить лимит пагинации).

Breaking changes — что ломает клиентов

Изменение Почему ломает Степень
Удаление поля из ответа Клиент ожидает поле, получает undefined — падает Критично
Переименование поля (name → fullName) Клиент ищет name, не находит Критично
Изменение типа поля (string → number) Клиент ожидает строку, получает число Критично
Изменение URL эндпоинта Клиент шлёт запрос на старый URL Критично
Изменение HTTP-метода (был GET, стал POST) Клиент использует GET Критично
Изменение статус-кода (был 200, стал 201) Клиент проверяет === 200 Средне
Добавление обязательного поля в запрос Старые клиенты его не шлют Критично
Изменение формата даты (DD.MM → ISO 8601) Клиент парсит по старому формату Критично
Добавление нового поля в ответ Клиент игнорирует неизвестные поля Безопасно

Как думает Senior. «Кажется, что переименовать поле — мелочь. Но за каждым полем в ответе стоит клиентский код: парсинг, отображение на UI, валидация, расчёты. Клиент не узнает о переименовании — он просто упадёт. Если очень нужно переименовать — добавьте новое поле, а старое пометьте deprecated. Через полгода можно удалить, предупредив всех клиентов.»

Как добавлять новые поля без боли

  1. Расширяйте, а не изменяйте. Новое поле — новый ключ в JSON. Старые поля остаются без изменений.
  2. Используйте дефолтные значения. Если поле опционально — укажите значение по умолчанию в документации.
  3. Помечайте deprecated. Устаревшие поля документируйте с пометкой deprecated и указывайте, чем их заменить.
  4. Дайте миграционный период. Удаляйте поля не раньше, чем через N месяцев после объявления deprecation.
  5. Документируйте. Любое изменение должно быть отражено в changelog API.

Типичная ошибка. «Мы удалим поле user_name, потому что оно нигде не используется на фронтенде». Но оно может использоваться мобильным приложением, которое обновляется раз в месяц, или сторонним партнёром, о котором вы забыли. Правило: если вы не знаете всех клиентов API — вы не можете просто удалить поле.

2.7. Идемпотентность — важное понятие

Идемпотентность звучит страшно, но объясняется просто: сколько раз ни повторяй запрос — результат будет одинаковым.

Представьте кнопку лифта. Если нажать один раз — лифт приедет. Если нажать пять раз — лифт всё равно приедет один раз. Кнопка лифта идемпотентна. А кнопка дверного звонка — нет: нажали один раз — звонок прозвенел один раз, нажали пять раз — прозвенел пять раз.

Почему это важно

В мире API сети ненадёжны. Клиент отправил запрос, но ответ не пришёл (таймаут, обрыв соединения). Что делать клиенту? Повторить запрос. Но если запрос не идемпотентен — повторное выполнение может создать дубликат: второй заказ, второе списание денег, вторую регистрацию.

Метод Идемпотентен? Комментарий
GET Да Просто читает данные
PUT Да Заменяет ресурс целиком. Повтор — то же состояние.
DELETE Да Первый раз удаляет, второй раз — 404, но это нормально.
PATCH Нет* Зависит от реализации. Если патч «прибавить 10 к balance» — не идемпотентен.
POST Нет Создаёт новый ресурс. Повтор — ещё один ресурс.

Идемпотентность в платежах

Платежи — критическая область, где отсутствие идемпотентности ведёт к финансовым потерям. Рассмотрим сценарий:

Клиент → POST /api/payments → Сервер списывает деньги ← [ТАЙМАУТ] ← Клиент не получил ответ Клиент → POST /api/payments → Сервер списывает деньги снова ← 200 OK ← Два списания!

Решение — Idempotency-Key:

Клиент → POST /api/payments Headers: { Idempotency-Key: "uuid-123" } → Сервер: проверяет, был ли ключ "uuid-123" → Нет: списывает деньги, сохраняет результат под ключом → Да: возвращает сохранённый результат, НЕ списывая повторно Повторный запрос с тем же ключом → сервер возвращает тот же ответ. Результат: деньги списаны один раз.
// Первый запрос — создаём платёж с уникальным ключом
POST /api/payments
Idempotency-Key: a1b2c3d4-e5f6-7890-abcd-ef1234567890

{
  "from": "account-123",
  "to": "account-456",
  "amount": 1000,
  "currency": "USD"
}

// Ответ: 201 Created
{
  "paymentId": "pay-001",
  "status": "completed",
  "amount": 1000
}

// Клиент не получил ответ, повторяет с тем же ключом
POST /api/payments
Idempotency-Key: a1b2c3d4-e5f6-7890-abcd-ef1234567890

// Тело запроса то же самое

// Ответ: 200 OK (не 201 — потому что ресурс уже существует)
// Но деньги НЕ списываются повторно!
{
  "paymentId": "pay-001",
  "status": "completed",
  "amount": 1000
}

Как думает Senior. «Idempotency-Key — это спасательный круг для надёжности. Он превращает неидемпотентный POST в идемпотентную операцию. Все операции, которые меняют финансовое состояние, должны поддерживать этот механизм. Срок хранения ключа — минимум 24 часа, чтобы клиент успел повторить запрос при любых сетевых проблемах.»

2.8. Пагинация, сортировка, фильтрация — стандарты

Редкий API обходится без списков данных. А если данных много — их нужно разбивать на страницы, сортировать и фильтровать. У аналитика должен быть готовый шаблон для этого.

Page-based vs Cursor-based пагинация

Характеристика Page-based Cursor-based
Как работает ?page=1&size=20 — указываете номер страницы ?cursor=abc123&size=20 — указываете последний элемент
Стабильность При добавлении новых записей страницы «съезжают» Новые записи не влияют на текущий курсор
Переход к странице N Просто: ?page=N Только последовательно: next / prev
Производительность на больших данных OFFSET замедляется на больших значениях Стабильна, зависит от индекса по полю курсора
Когда использовать Небольшие наборы (<10K), UI с номерами страниц Ленты, бесконечный скролл, большие объёмы, реальное время
Пример Админка, список пользователей Лента новостей, история операций

Антипаттерн: offset-based пагинация в ленте. Представьте ленту новостей. Вы открываете страницу 1 — видите 20 постов. Пока вы читаете, приходит 5 новых постов. Вы нажимаете «страница 2» — и видите те же посты, что и на странице 1, потому что они «съехали» на 5 позиций. Пользователь думает, что сошёл с ума. Cursor-based пагинация решает эту проблему: курсор — это ID последнего поста, который вы видели. Новые посты добавляются сверху, но курсор остаётся на месте.

Стандартные параметры

Параметр Описание Пример
page Номер страницы (1-based) ?page=1
size Количество элементов на странице ?size=20
offset Смещение от начала (альтернатива page) ?offset=0&limit=20
sort Поле и направление сортировки ?sort=date,desc
filter или status Фильтрация по полю ?status=paid
q Полнотекстовый поиск ?q=iphone
cursor Идентификатор для cursor-based пагинации ?cursor=eyJsYXN0SWQiOjEwMH0=

Стандартный формат ответа с пагинацией

GET /api/v1/orders?page=1&size=20&sort=date,desc&filter=status:paid

{
  "data": [
    { "id": 1, "status": "paid", "total": 1500, "date": "2026-07-10T14:30:00Z" },
    { "id": 2, "status": "paid", "total": 2300, "date": "2026-07-09T10:15:00Z" }
  ],
  "pagination": {
    "page": 1,
    "size": 20,
    "totalItems": 58,
    "totalPages": 3,
    "hasNext": true,
    "hasPrev": false
  }
}

Как думает Senior: выбор стратегии пагинации.

«Для админки интернет-магазина (сотни заказов в день) — page-based пагинация идеальна. Пользователи хотят перейти к странице 5, администратору нужны точные номера страниц. Cursor-based здесь избыточен.

Для ленты операций в банковском приложении (миллионы транзакций) — только cursor-based. Номера страниц не нужны, а стабильность важна: между запросами могут пройти часы, за которые появятся тысячи новых операций.

Всегда устанавливайте лимит на size. Если клиент запросит size=100000 — верните ошибку 422 или обрежьте до разумного максимума (например, 100). Иначе один неаккуратный клиент положит базу данных.»

Пример комплексного запроса

# Запрос: заказы со статусом "paid" за последний месяц, 
# отсортированные по сумме (по убыванию), страница 2, 50 записей

GET /api/v1/orders?status=paid&dateFrom=2026-06-14&dateTo=2026-07-14
     &sort=total,desc&page=2&size=50

2.9. REST API в Confluence — как должен выглядеть контракт

Аналитик — не разработчик и не инженер документации. Но именно аналитик чаще всего описывает API в Confluence или другом инструменте. Вот минимальный набор информации, который должен быть у каждого эндпоинта.

Шаблон описания эндпоинта

Для каждого эндпоинта документация должна содержать:

  1. URL и метод — однозначная идентификация.
  2. Назначение — что делает эндпоинт на языке бизнеса.
  3. Параметры запроса — path, query, headers, body.
  4. Пример запроса — curl или JSON.
  5. Пример ответа — успешный сценарий.
  6. Возможные статус-коды — все, не только 200.
  7. Возможные ошибки — описание ошибок для каждого статуса.
  8. Ограничения — rate limit, максимальный размер, пагинация.

Пример готового описания эндпоинта

Создание заказа

Метод POST
URL /api/v1/orders
Назначение Создание нового заказа. Заказ создаётся из товаров, добавленных в корзину пользователя. После создания корзина очищается.
Аутентификация Требуется Bearer token (заголовок Authorization)
Idempotency-Key Обязателен в заголовке. Повторный запрос с тем же ключом не создаёт дубликат.

Request Body

{
  "cartId": "cart-42",            // string, required — ID корзины пользователя
  "shippingAddressId": "addr-7",  // string, required — ID адреса доставки

  "comment": "Позвонить перед доставкой", // string, optional — комментарий к заказу
  "deliveryDate": "2026-07-20"    // string (date), optional — желаемая дата доставки
}

Код Тело ответа

{
  "orderId": "ORD-20260714-001",

  "items": [

Возможные ошибки

Код Тело ошибки Когда возникает
400 Bad Request {"error": "validation_error", "details": [{"field": "cartId", "message": "cartId is required"}]} Не указан cartId, невалидный JSON
401 Unauthorized {"error": "unauthorized", "message": "Missing or invalid token"} Токен не предоставлен или истёк
404 Not Found {"error": "not_found", "message": "Cart cart-42 not found"} Корзина или адрес не найдены
409 Conflict {"error": "conflict", "message": "Idempotency-Key already used"} Повторный запрос с тем же Idempotency-Key, но другим телом
422 Unprocessable Entity {"error": "business_error", "message": "Cart is empty"} Корзина пуста, товара нет в наличии, сумма меньше минимальной
429 Too Many Requests {"error": "rate_limit", "message": "Too many requests. Try again in 30 seconds"} Превышен лимит запросов (100 запросов в минуту)
500 Internal Server Error {"error": "internal_error", "message": "Something went wrong"} Внутренняя ошибка сервера

Лучшая практика. Добавьте в документацию раздел с примерами ошибок. Разработчики клиентской стороны (фронтенд, мобильное приложение) будут вам благодарны: они увидят не только код, но и формат тела ошибки, что позволяет правильно обработать каждый сценарий.

Чек-лист: 10 вопросов, которые аналитик должен задать при проектировании API

  1. Какой ресурс? Какая сущность предметной области лежит в основе эндпоинта?
  2. Какой HTTP-метод? GET, POST, PUT, PATCH или DELETE — и почему?
  3. Какой URL? Это существительное во множественном числе? Нет ли глагола в пути?
  4. Какие статус-коды? Каждый возможный исход операции имеет свой код?
  5. Нужна ли идемпотентность? Может ли запрос быть повторён? Есть ли Idempotency-Key?
  6. Какая пагинация? Page-based или cursor-based? Какие параметры?
  7. Какая фильтрация и сортировка? Какие поля можно фильтровать? Какой формат sort?
  8. Что с обратной совместимостью? Не ломает ли новое поле существующих клиентов?
  9. Какие ошибки? Описан ли формат ошибки для каждого статус-кода?
  10. Как документируется? Есть ли пример запроса и ответа для каждого сценария?

Что должен понять аналитик

  • REST API — это не магия, а строгие соглашения о том, как клиент и сервер общаются.
  • Ресурсы (существительные) — основа REST. Методы (глаголы) — способ манипуляции ресурсами.
  • URL должен быть читаемым, предсказуемым и единообразным.
  • Статус-коды — это язык, который клиент понимает без разбора тела ответа.
  • Контракт API менять нельзя без миграционного периода.
  • Идемпотентность — база надёжных финансовых операций.
  • Документация API — это не «для галочки», а инструмент, который экономит часы переписки с разработчиками.

Золотое правило. Хороший API — это API, который не заставляет клиента читать документацию, чтобы понять, как им пользоваться. А если документация всё-таки нужна — она должна быть настолько полной, чтобы клиент ни разу не задал вопрос разработчику.
