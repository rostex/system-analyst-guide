Глава 8. Производительность

Почему «просто добавить поле» может убить базу и как аналитику научиться считать стоимость операций

Введение

Представьте, что производительность системы — это личный бюджет. У вас есть 1000 рублей в месяц на все расходы. Простой запрос к базе данных стоит 1 рубль. Сложный — 100 рублей. Если вы не следите за расходами, к 15-му числу вы останетесь без денег.

В разработке то же самое. Каждый запрос, каждая операция, каждое решение имеет свою «цену». Разработчики платят эту цену процессорным временем, памятью, дисковыми операциями. Аналитик, который не понимает цены операций, будет формулировать требования, которые невозможно выполнить в рамках доступных ресурсов.

Эта глава — про то, как научиться видеть стоимость операций и принимать осознанные решения. Вы не станете администратором баз данных, но начнёте понимать, почему разработчик говорит: «Этот запрос убьёт прод» — и почему к нему стоит прислушаться.

Факт: 70% проблем с производительностью в продакшене закладываются на этапе написания требований. Аналитик, который понимает производительность, — самый ценный участник команды.

7.1. Откуда берутся тормоза

Прежде чем говорить о конкретных проблемах, давайте разберёмся, из чего вообще складывается время ответа системы. Каждая операция в компьютере имеет свою стоимость, и разница между ними — на порядки.

Сеть (Network Latency)

Когда один сервис обращается к другому по сети, каждый вызов стоит времени на передачу данных. Внутри одного дата-центра задержка составляет 0.5–2 мс. Между континентами — 100–300 мс. Если сервис делает 50 сетевых вызовов, чтобы собрать один ответ, — это уже 50 × 2 мс = 100 мс только на сеть.

CPU (Процессор)

Самый быстрый ресурс. Процессор выполняет миллиарды операций в секунду. Но если алгоритм неэффективен (например, вложенные циклы по 100 000 элементов), даже процессор начинает «тормозить».

Память (RAM)

Доступ к оперативной памяти в 100 раз медленнее, чем выполнение одной инструкции процессора. Если данные не помещаются в RAM — система начинает использовать диск, и тут начинаются настоящие проблемы.

Диск (Disk I/O)

Самое узкое место. Чтение с SSD — около 0.1–0.2 мс. Чтение с HDD — 5–10 мс. Кажется, что это мелочь, но когда база данных читает миллион строк с диска, это превращается в секунды и десятки секунд.

База данных (Database)

База — самый частый bottleneck в корпоративных системах. Она совмещает в себе всё: чтение с диска, использование CPU для обработки запросов, память для кэширования. И она же — единственное место, где хранятся все данные. Ошибка в запросе может заблокировать таблицу на минуты.

ОперацияВремяМасштаб

RAM (оперативная память)~100 нс330

Запомните: Разница между обращением к RAM и к HDD — 100 000 раз. То, что в памяти читается за 1 секунду, с диска будет читаться больше суток.

7.2. N+1 проблема — главный враг

Самая распространённая проблема производительности, которую аналитик может предотвратить ещё на этапе требований, — это N+1 запросов.

Что такое N+1

Ситуация: вы делаете один запрос, чтобы получить список сущностей (например, заказов), а затем для каждой сущности делаете отдельный запрос, чтобы получить связанные данные (товары в заказе).

Если у вас 100 заказов, вы выполните 1 + 100 = 101 запрос вместо одного.

Клиент Сервер База данных | | | |-- 1. Список заказов ->| | | |-- SELECT * FROM orders ->| | |<-- 100 заказов ----------| | | | | |-- SELECT * FROM items | | | WHERE order_id = 1 -> | | |<-- товары заказа 1 -----| | | | | |-- SELECT * FROM items | | | WHERE order_id = 2 -> | | |<-- товары заказа 2 -----| | | ... | | |-- SELECT * FROM items | | | WHERE order_id = 100 ->| | |<-- товары заказа 100 ---| |<-- данные -----------| |

Как выглядит в коде (псевдокод)

// ❌ N+1 проблема
orders = db.query("SELECT * FROM orders WHERE user_id = 42")
for order in orders:
    items = db.query("SELECT * FROM items WHERE order_id = " + order.id)
    // обрабатываем каждый заказ

// ✅ Решение: один JOIN
orders = db.query("""
    SELECT o.*, i.*
    FROM orders o
    JOIN items i ON i.order_id = o.id
    WHERE o.user_id = 42
""")

❌ Ошибка аналитика: В требованиях не указано, какие связанные данные нужны. Разработчик реализует «ленивую загрузку» (Lazy Loading) — данные подтягиваются по мере обращения. В результате — N+1 запросов в рантайме.

✅ Решение: Всегда указывайте в требованиях: «При загрузке списка заказов должны сразу загружаться: список товаров, статус доставки, информация о плательщике». Это называется Eager Loading («жадная загрузка») — все данные одним запросом.

Способы решения

  • JOIN — получение связанных данных одним SQL-запросом
  • Batch Loading — собрать все identity, потом одним IN-запросом получить все связанные записи
  • Eager Loading — явное указание ORM-фреймворку, что нужно загрузить

🧠 Senior: Аналитик должен явно прописывать граф загрузки (loading graph). Если в требованиях написано «отобразить список заказов», этого мало. Нужно: «отобразить список заказов с превью первого товара, суммой, статусом и датой доставки». Тогда разработчик точно знает, какие JOIN делать.

7.3. Индексы — как они влияют на скорость

Индексы в базе данных работают как предметный указатель в книге. Если вам нужно найти все упоминания слова «производительность» в книге на 500 страниц, вы можете:

  • Прочитать все 500 страниц (Full Table Scan — 500 страниц)
  • Открыть указатель, найти слово, перейти на нужные страницы (Index Scan — 5 страниц)

Типы сканирования

ТипЧто делаетСкоростьКогда используется
Full Table ScanЧитает все строки таблицыОчень медленно на больших таблицахНет подходящего индекса; таблица маленькая
Index ScanИдёт по B-дереву индекса, находит нужные строкиБыстро (логарифмическая сложность)Есть индекс, условие селективно
Index Only ScanЧитает данные прямо из индекса, не обращаясь к таблицеМаксимально быстроВсе нужные поля есть в индексе

Пример: как индекс меняет план запроса

-- Таблица: orders (10 000 000 строк)
-- Есть индекс на status

-- ❌ Без индекса (Full Table Scan)
EXPLAIN SELECT * FROM orders WHERE status = 'active';
-- Seq Scan on orders  (cost=0.00..180320.00 rows=5000000)
--   Filter: (status = 'active'::text)

-- ✅ С индексом (Index Scan)
EXPLAIN SELECT * FROM orders WHERE status = 'active';
-- Index Scan using idx_orders_status on orders
--   (cost=0.56..34215.00 rows=5000000)
--   Index Cond: (status = 'active'::text)

Факт: Full Table Scan на таблице в 10 млн строк может выполняться 10–30 секунд. Index Scan — 0.1–0.5 секунды. Разница в 100 раз.

Селективность

Селективность — это доля строк, которые отбирает условие WHERE. Чем меньше строк отбирается, тем выше селективность и тем эффективнее индекс.

  • WHERE status = 'deleted' — отбирает 0.1% строк → высокая селективность → индекс эффективен
  • WHERE status IS NOT NULL — отбирает 99% строк → низкая селективность → индекс бесполезен

❌ Типичная ошибка: Использование функции в WHERE. Индекс на поле date не сработает, если написать WHERE YEAR(date) = 2024. База выполняет функцию для каждой строки, индекс не используется.

-- ❌ Индекс не используется
SELECT * FROM orders WHERE YEAR(created_at) = 2024;

-- ✅ Индекс используется (диапазонное условие)
SELECT * FROM orders WHERE created_at >= '2024-01-01'
                        AND created_at < '2025-01-01';

🧠 Senior: Аналитик должен знать, какие поля будут в WHERE, ORDER BY и JOIN. Эти поля должны быть проиндексированы. Если требований много и они разные — разработчик не знает, какие индексы строить. Задача аналитика — выделить «горячие» пути запросов и явно указать: «поиск всегда по статусу и дате».

7.4. JOIN — плата за связность

Каждый JOIN в SQL-запросе — это дополнительная работа для базы данных. База должна соединить строки из двух (или больше) таблиц по условию, и это стоит ресурсов.

Стоимость JOIN

Сложность JOIN зависит от алгоритма, который выбирает база:

АлгоритмСложностьКогда используется
Nested Loop JoinO(N × M)Одна таблица маленькая, вторая большая
Hash JoinO(N + M)Большие таблицы, нет индекса
Merge JoinO(N log N + M log M)Данные отсортированы

Каждый дополнительный JOIN может увеличивать время выполнения в 2–5 раз, особенно если таблицы большие.

🏦 Пример: Отчёт по продажам с 8 JOIN на таблицах по 2–5 млн строк выполнялся 30 секунд. После оптимизации (добавление индексов, уменьшение количества JOIN через денормализацию) время упало до 2 секунд.

JOIN vs Денормализация

ПодходПлюсыМинусы
Нормализация + JOINНет дублирования, консистентностьМедленные отчёты, сложные запросы
ДенормализацияБыстрое чтение, простые запросыДублирование, сложность обновления

Совет: Для OLTP-систем (операционные — заказы, платежи) — нормализация и минимум JOIN. Для OLAP-систем (аналитические — отчёты, дашборды) — допустима денормализация и много JOIN.

7.5. Пагинация и её цена

Проблема OFFSET

Самая популярная реализация пагинации — OFFSET + LIMIT — имеет скрытую проблему. Чем больше OFFSET, тем медленнее запрос.

-- Страница 1: быстро
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 0;

-- Страница 10 000: очень медленно
SELECT * FROM orders ORDER BY id LIMIT 20 OFFSET 199980;

-- Почему? База всё равно читает 200 000 строк,
-- просто возвращает последние 20

Cursor-Based пагинация

Решение — пагинация на основе курсора (ключа). Вместо номера страницы передаём идентификатор последнего элемента.

-- Курсорная пагинация: всегда быстро
SELECT * FROM orders
WHERE id > 199980
ORDER BY id
LIMIT 20;

-- Неважно, какой курсор — всегда читаем 20 строк
Тип пагинацииСкорость на странице 1Скорость на странице 10 000
OFFSET + LIMIT0.01 с5.2 с
Cursor-based0.01 с0.01 с

⚠️ Внимание: Требование «выгрузить все 10 000 записей на одну страницу» — это архитектурная проблема. Никогда не формулируйте такое в требованиях. Если пользователю нужны все данные — используйте фоновую выгрузку (CSV, Excel, async task).

🏦 Пример: Аналитик согласовал требование «скачать отчёт за год в Excel». Разработчик реализовал выгрузку в потоке запроса. При 2 млн строк — таймаут через 60 секунд, сервер упал с OutOfMemoryError. Правильное решение: асинхронная генерация через очередь задач (RabbitMQ / Kafka).

7.6. Ограничения на стороне API

Даже если ваша база данных справляется, внешние ограничения могут «убить» производительность. Аналитик должен учитывать:

Rate Limiting (Ограничение частоты запросов)

Любой публичный API имеет лимит: 100 запросов в минуту, 10 000 в час и т.д. Если ваша система в цикле делает запросы к внешнему API — вы упрётесь в лимит. Требование «получать курсы валют каждые 5 секунд» может быть нереализуемо, если внешний сервис даёт максимум 1 запрос в минуту.

Timeout (Таймаут)

Каждый запрос к сервису имеет таймаут — максимальное время ожидания ответа. Если запрос не укладывается — клиент получает ошибку 504 Gateway Timeout.

  • Обычный таймаут: 30 секунд
  • Для отчётов: 2–5 минут
  • Для файловых выгрузок: вообще не в HTTP-запросе, а через асинхронные задачи

Payload Size (Размер ответа)

Передача 10 МБ JSON с сервера клиенту — это:

  • 10 МБ трафика
  • Время сериализации на сервере
  • Время десериализации на клиенте
  • Память на обеих сторонах
  • Замедление рендера интерфейса

🧠 Senior: Перед тем, как писать требование «API возвращает список всех заказов», оцените максимальный размер ответа. 1 заказ = 2 КБ. Заказов 10 000. Ответ = 20 МБ. Это неприемлемо. Нужна пагинация.

7.7. Кэширование — что это и как работает

Кэш — это быстрый слой хранения между приложением и базой данных. Вместо того, чтобы каждый раз ходить в базу (медленно), приложение сначала проверяет кэш (быстро).

Запрос клиента | v [Кэш] ──есть данные?──> Вернуть из кэша (1 мс) | нет v [База данных] ──> Сохранить в кэш ──> Вернуть (100 мс)

Redis / Memcached

Популярные in-memory кэши хранят данные в оперативной памяти. Redis — самый распространённый: поддерживает строки, хэши, списки, очереди, TTL. Memcached — проще, только ключ-значение.

Cache Invalidation — самая сложная проблема

В программировании есть две сложные проблемы: инвалидация кэша, именование и ошибка на единицу. Инвалидация кэша — это проблема «когда обновить данные в кэше, если данные в базе изменились».

  • TTL (Time To Live): данные живут N секунд, потом автоматически удаляются. Просто, но данные могут быть устаревшими.
  • Write-Through: при каждом обновлении БД пишем и в кэш. Актуально, но больше операций записи.
  • Eventual Invalidation: при изменении данных отправляем событие, подписчик чистит кэш. Сложно, но гибко.

💡 Пример: Каталог товаров на сайте — идеальный кандидат для кэширования. Товары меняются раз в час, но читаются тысячу раз в минуту. Кэшируем на 5 минут (TTL = 300). База отдыхает.

❌ Ошибка: Кэширование данных, которые часто меняются. Например, баланс счёта пользователя. Если закэшировать баланс на 5 минут — пользователь увидит неверный баланс, начнутся проблемы с оплатой. Такие данные кэшировать нельзя.

7.8. Batch Processing — обработка пачками

Одна из самых простых и эффективных оптимизаций — не делать запросы по одному, а собирать их в пачки (batch).

Пример: вставка 1000 записей

-- ❌ 1000 отдельных INSERT
INSERT INTO log (msg) VALUES ('entry 1');
INSERT INTO log (msg) VALUES ('entry 2');
-- ... 998 раз ...
INSERT INTO log (msg) VALUES ('entry 1000');
-- Время: ~10 секунд (каждый INSERT — отдельная транзакция)

-- ✅ Один batch-INSERT
INSERT INTO log (msg) VALUES
  ('entry 1'),
  ('entry 2'),
  -- ...
  ('entry 1000');
-- Время: ~0.1 секунды (одна транзакция)

Факт: Разница между вставкой 1000 записей по одной и одной пачкой — в 100 раз. Одна транзакция вместо тысячи. Одна коммит-запись на диск вместо тысячи.

Batch Processing в отчётах

Генерация Excel-отчёта на 100 000 строк:

  • В цикле, строка за строкой — 5 минут и 50 МБ RAM
  • Пачками по 10 000 строк — 30 секунд и 10 МБ RAM

🧠 Senior: Если данных много — нужна асинхронная обработка. Никогда не генерируйте большие отчёты в HTTP-запросе. Пользователь нажимает «Скачать», сервер ставит задачу в очередь, пользователь получает уведомление, когда отчёт готов. Так делают все серьёзные системы.

7.9. Как аналитик влияет на производительность

Вы, как аналитик, оказываете наибольшее влияние на производительность системы — ещё до того, как написана первая строка кода. Ваши требования напрямую определяют нагрузку.

Пример 1: «Показывать историю заказов за 5 лет»

У компании 50 000 заказов в месяц. За 5 лет — 3 млн заказов. Если открывать страницу со всеми заказами за 5 лет — 3 млн записей, 2–3 секунды загрузки, 50 МБ трафика.

Что должен спросить аналитик: «Действительно ли пользователю нужны все 5 лет сразу? Может, достаточно года с возможностью посмотреть архив?»

Пример 2: «Отчёт должен обновляться в реальном времени»

Если дашборд перезапрашивает данные каждые 5 секунд, а запрос выполняется 2 секунды — база не успевает отдыхать. 12 запросов в минуту × 8 часов = 5760 запросов в день только на один отчёт.

Что должен спросить аналитик: «Какая периодичность обновления действительно нужна бизнесу? Может, раз в минуту или раз в час?»

Пример 3: «Проверять дубликаты при каждом сохранении»

При сохранении каждого документа система проверяет: есть ли уже документ с таким номером. Если таблица большая (10 млн строк), а проверка — Full Table Scan, то каждое сохранение будет стоить 5 секунд.

Что должен спросить аналитик: «Как часто возникают дубликаты? Если редко — может, проверять раз в день фоновой задачей?»

✅ Золотое правило: Прежде чем включить требование в спецификацию, задайте себе три вопроса:

1. Сколько данных будет обработано за один раз?

2. Как часто эта операция будет выполняться?

3. Можно ли сделать это не в реальном времени, а асинхронно?

Итоги главы

Чек-лист: вопросы о производительности в требованиях

  • Сколько записей будет обрабатывать этот запрос? (100, 10 000, 10 млн?)
  • Какие связанные данные нужно загружать вместе с основным объектом?
  • По каким полям будет выполняться поиск и сортировка?
  • Какая периодичность выполнения — раз в секунду или раз в день?
  • Сколько одновременно пользователей будут выполнять эту операцию?
  • Нужна ли пагинация? Какой максимальный размер страницы?
  • Можно ли кэшировать результат? Какой TTL допустим?
  • Можно ли выполнять операцию асинхронно, а не в реальном времени?
  • Какой максимальный размер ответа (payload) допустим?
  • Какие внешние системы задействованы? Есть ли у них rate limits?

Что должен понять аналитик

  • Каждая операция имеет «цену»: время, память, ресурсы.
  • N+1 проблема — самый частый источник тормозов.
  • Индексы ускоряют запросы, но требуют понимания, какие поля будут в WHERE.
  • JOIN — это плата за связность данных.
  • OFFSET-пагинация деградирует с ростом номера страницы.
  • Кэширование спасает, но требует стратегии инвалидации.
  • Batch-обработка в 100 раз быстрее последовательных запросов.
  • Аналитик — первый, кто может предотвратить проблему производительности.

  1. Почему запрос OFFSET 100000 LIMIT 20 выполняется так же долго, как запрос без LIMIT?

Запомните: Производительность — это не задача разработчика. Это задача всей команды. Аналитик, который понимает «цену» каждой операции, — лучший защитник системы от перегрузок.
