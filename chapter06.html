Введение

Представьте, что вы строите дом. Вы тщательно продумываете планировку, выбираете материалы, нанимаете архитектора. Но если фундамент залит с нарушением технологии — дом пойдёт трещинами. Никакой ремонт не исправит кривой фундамент, кроме полного сноса и перестройки.

База данных — это фундамент вашего приложения. Все фичи, весь бизнес-логика, вся аналитика — всё опирается на данные. И если схема базы данных спроектирована плохо, любая новая функциональность будет даваться с болью. Интерфейс можно переписать за месяц. Бэкенд — за квартал. Миграция данных с переездом на новую схему — это полгода ада и потерянные деньги.

Цель этой главы — не сделать из вас администратора баз данных. Вы не будете настраивать кластер PostgreSQL или оптимизировать буферный кэш. Но вы начнёте понимать последствия решений, которые принимает разработчик (или вы сами) на уровне схемы. Вы научитесь замечать ошибки, которые аукнутся через год, и будете говорить с командой на одном языке.

Запомните: Аналитик, который понимает БД, — это аналитик, которому доверяют разработчики. Потому что он перестаёт предлагать "просто добавить поле" и начинает мыслить схемами, индексами и целостностью данных.

5.1. Таблицы — это не просто "таблички в Excel"

На первый взгляд таблица базы данных похожа на лист Excel: строки, колонки, ячейки. Но сходство обманчиво. Между Excel и таблицей в реляционной БД — пропасть.

Строка = запись, колонка = атрибут

В Excel вы можете записать в ячейку что угодно: текст, число, дату, формулу, картинку. В БД каждая колонка имеет строгий тип. Нельзя положить строку в колонку price DECIMAL. База данных вас просто не пустит, вернув ошибку.

Каждая строка — это запись (entity). Каждая колонка — атрибут (attribute) этой сущности. Строка не может быть "разношёрстной": если в колонке email первой строки текст, а второй — число, это ошибка проектирования в Excel и невозможно в БД.

Схема — структура, а не данные

Разница в подходе:

  • Excel: вы открываете файл и видите данные. Структура (заголовки) — это просто первая строка. Вы можете вставить колонку посередине и ничего не сломается.
  • БД: сначала вы описываете структуру (схему) — таблицу, колонки, типы, ограничения. Потом наполняете данными. Изменить схему — это операция ALTER TABLE, которая может заблокировать таблицу на часы.
Сравнение: Excel: ┌──────────┬───────────┬──────────┐ │ Имя │ Возраст │ Город │ ← это строка, а не схема ├──────────┼───────────┼──────────┤ │ Иван │ 30 │ Москва │ │ Петя │ двадцать │ 12345 │ ← разнотипные данные └──────────┴───────────┴──────────┘ SQL (схема): CREATE TABLE users ( name VARCHAR(100) NOT NULL, age INTEGER NOT NULL, city VARCHAR(100) NOT NULL ); ──── тип каждой колонки задан ЖЁСТКО ────

Почему тип колонки — это критично?

Тип данных — это контракт. Когда вы объявляете price DECIMAL(10,2), база данных гарантирует:

  • В колонке только числа с двумя знаками после запятой
  • Над колонкой работают математические операции
  • Индексы сортируют числовые значения корректно
  • Любой клиент (Python, Java, Go) знает, как интерпретировать значение

Метафора: Таблица БД — это анкета со строгими правилами заполнения. Поле "Возраст" принимает только числа от 0 до 150. Поле "Email" проверяет формат. Вы не можете написать "двадцать пять" в поле "Возраст" — анкета не примется. В Excel вы можете написать что угодно. В этом сила БД.

5.2. Первичный ключ (Primary Key)

Первичный ключ (PK) — это колонка (или набор колонок), которая уникально идентифицирует каждую строку в таблице. Если у таблицы нет PK — вы не можете гарантировать, что две строки не окажутся одинаковыми.

Surrogate vs Natural ключи

Есть два принципиально разных подхода к выбору первичного ключа:

ХарактеристикаSurrogate (суррогатный)Natural (естественный)
ЗначениеИскусственное, не имеет бизнес-смыслаБерётся из предметной области
Примерыid INTEGER AUTO_INCREMENT, UUIDНомер паспорта, Email, SNILS, VIN автомобиля
Меняется ли?Никогда (это просто число)Может — человек сменил паспорт, email, машину продали
РекомендацияПочти всегдаПочти никогда

Ошибка: Использовать номер паспорта как Primary Key. Человек сменил паспорт — вы должны обновить ключ. Но на него ссылаются внешние таблицы! Придётся каскадно обновлять тысячи записей. Никогда не используйте Natural ключи как Primary Key, если есть хоть малейший шанс, что значение изменится.

UUID vs Auto Increment

КритерийAUTO_INCREMENTUUID
Размер4 байта (INT) или 8 байт (BIGINT)16 байт (или 36 символов как строка)
Скорость вставкиБыстро (sequential)Медленнее (random — фрагментация индекса)
БезопасностьМожно угадать ID (инкремент)Нельзя угадать
Распределённые системыКонфликтыУникален глобально
ЧеловекочитаемостьДа (легко отслеживать)Нет (сплошная каша)

Лучшая практика: В монолитном приложении с одним сервером БД — используйте BIGINT AUTO_INCREMENT. В микросервисах, распределённых системах или когда ID создаются на клиенте — UUIDv7 (time-ordered UUID, не фрагментирует индекс). UUIDv4 избегайте — он убивает производительность индексов.

Составные ключи

Иногда одной колонки недостаточно. Например, таблица связи "товар — корзина":

-- Каждый товар может быть в корзине только один раз
CREATE TABLE cart_items (
    cart_id    INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    quantity   INTEGER NOT NULL DEFAULT 1,
    PRIMARY KEY (cart_id, product_id)
);

Здесь первичный ключ состоит из двух колонок. Это гарантирует, что один и тот же товар не появится в корзине дважды. И заодно ссылается на внешние ключи cart_id и product_id.

Ошибка: Нет Primary Key. Можно вставить дубликат, и никто ничего не заметит. Когда система вырастет, почистить дубликаты будет мучительно больно.

-- Таблица без PK — беспорядок гарантирован
CREATE TABLE log (
    message TEXT,
    level   VARCHAR(10),
    ts      TIMESTAMP
);
-- Две одинаковые записи? Пожалуйста:
INSERT INTO log VALUES ('hello', 'INFO', NOW());
INSERT INTO log VALUES ('hello', 'INFO', NOW());
-- Никакой ошибки. А теперь найдите, какая запись "лишняя".

5.3. Внешние ключи (Foreign Key)

Внешний ключ (FK) — это ссылка одной таблицы на другую. Если у вас есть таблица orders и таблица customers, то заказ должен принадлежать реальному клиенту. FK обеспечивает ссылочную целостность.

CREATE TABLE customers (
    id   BIGINT PRIMARY KEY,
    name VARCHAR(200) NOT NULL
);

CREATE TABLE orders (
    id          BIGINT PRIMARY KEY,
    customer_id BIGINT NOT NULL,
    total       DECIMAL(18,2) NOT NULL,
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

Теперь нельзя создать заказ с customer_id = 42, если клиента с таким ID не существует. База данных проверит и вернёт ошибку.

Что будет, если FK нет?

Появятся "сиротские" записи — заказы, которые ссылаются на несуществующих клиентов. Это происходит, если разработчик удалил клиента, но забыл удалить его заказы. Или если данные загружались из разных источников без проверки связей.

Банковский пример: В таблице transactions есть account_id, который ссылается на accounts.id. Если FK нет, может возникнуть ситуация: транзакция есть, а счёт удалён. Куда ушли деньги? Никто не знает. Аудит не сойдётся. Регулятор наложит штраф.

CASCADE, SET NULL, RESTRICT — что выбрать?

При удалении записи из родительской таблицы нужно решить, что делать с дочерними:

ПравилоЧто происходитКогда использовать
RESTRICT / NO ACTIONЗапретить удаление, если есть дочерние записиБезопасный режим по умолчанию
CASCADEУдалить все дочерние записиЗависимые сущности — например, удаление пользователя удаляет его сессии
SET NULLОбнулить внешний ключ в дочерних записяхНеобязательная связь — удаление отдела оставляет сотрудников без отдела

Предупреждение: CASCADE опасен каскадным удалением. DELETE FROM customers WHERE id = 1 может неожиданно удалить тысячи заказов, позиций заказов и логов. Всегда проверяйте, что будет затронуто. В некоторых системах CASCADE вообще запрещён на уровне code review.

Ошибка: Удаление клиента с заказами. Если FK нет — заказы становятся сиротами. Если FK есть с CASCADE — заказы удаляются без предупреждения. Если FK есть с RESTRICT — DELETE вернёт ошибку, и разработчик будет вынужден решить: архивировать заказы или запретить удаление. Это правильное поведение.

Почему разработчики иногда убирают FK?

В高性能 системах (high-load) проверка внешних ключей создаёт нагрузку на БД при каждой вставке и обновлении. Некоторые команды сознательно убирают FK, перенося контроль целостности в код приложения. Это компромисс: скорость против надёжности.

Senior: Если вы убираете FK ради производительности, вы берёте на себя ответственность за целостность данных. В коде приложения не должно быть ни одного бага, который создаст сиротскую запись. Ошибка — и данные сломаны. Только опытные команды с высоким покрытием тестами могут позволить себе это. Для 95% проектов FK обязательны.

5.4. Типы данных — на что обратить внимание

Выбор типа данных — не техническая деталь, а бизнес-решение. Тип определяет, какие значения допустимы, сколько места занимает колонка, как она индексируется и как с ней работают запросы.

VARCHAR(N) vs TEXT

ТипКогда использоватьПример
VARCHAR(50)Короткие строки с известной макс. длинойEmail, телефон, имя
VARCHAR(255)Стандартный выбор для недлинных строкАдрес, название компании
TEXTДлинный текст без ограничения по длинеОписание товара, комментарий, статья в блоге

Факт: В большинстве СУБД (PostgreSQL, MySQL с InnoDB) разница в производительности между VARCHAR(N) и TEXT незначительна для колонок до нескольких килобайт. Главное — выбирать осмысленный лимит: VARCHAR(2000) для адреса — это странно, а VARCHAR(254) для email — обоснованно.

INTEGER vs BIGINT

Кажется, что выбор очевиден — бери BIGINT всегда. Но есть нюансы:

ТипДиапазонРазмерРиск переполнения
INTEGER (INT)−231 … +231−1 (≈2.1 млрд)4 байтаЕсть — для ID популярных таблиц
BIGINT−263 … +263−1 (≈9.2 квинтиллиона)8 байтПрактически нет
SMALLINT−32 768 … +32 7672 байтаВысокий — для статусов, флагов

Ошибка: Использовать INTEGER для ID в таблице, которая будет расти годами. Счётчик переполнится, и вы не сможете вставить новую строку. История знает случаи, когда production падал, потому что AUTO_INCREMENT дошёл до лимита. Берите BIGINT для ID всегда.

DECIMAL(18,2) — деньги

Деньги — критически важный тип данных. Никогда не используйте FLOAT или DOUBLE для денег. Почему?

-- Проблема с float:
SELECT 0.1 + 0.2;  -- Результат: 0.30000000000000004

-- Правильно с decimal:
SELECT 0.1::decimal + 0.2::decimal;  -- Результат: 0.30

Правило: для денег — DECIMAL(18,2) (или DECIMAL(18,4) для валют с четырьмя знаками после запятой). 18,2 означает: всего 18 цифр, из них 2 — после запятой. Этого хватит для сумм до 999 триллионов с копейками.

TIMESTAMP vs DATETIME

ТипДиапазонЧасовой поясРекомендация
TIMESTAMP1970…2038 (Unix epoch)Учитывает timezonePostgreSQL — да, MySQL — аккуратно
DATETIME1000…9999Не учитываетЕсли храните уже в UTC
TIMESTAMPTZ (PG)1970…9999Учитывает timezoneЛучший выбор для PostgreSQL

Лучшая практика: Храните все временные метки в UTC. Преобразование в локальный часовой пояс делайте на стороне приложения или в представлениях (views). Используйте TIMESTAMP WITH TIME ZONE в PostgreSQL.

BOOLEAN — есть не везде

В PostgreSQL есть полноценный тип BOOLEAN (true/false). В MySQL его нет — используется TINYINT(1) (0/1). В SQLite — INTEGER (0/1). Аналитик должен знать: флаг — это не всегда "галочка", это может быть 0 в базе, что в коде превращается в false.

ENUM — удобно, но больно менять

ENUM кажется удобным для полей с фиксированным набором значений:

CREATE TABLE orders (
    status ENUM('pending', 'paid', 'shipped', 'delivered', 'cancelled')
);

Проблема: если нужно добавить новое значение (например, 'refunded'), придётся выполнять ALTER TABLE ... MODIFY COLUMN, что может заблокировать таблицу. Альтернатива: отдельная таблица-справочник order_statuses с FK.

JSON — удобно, но антипаттерн

Современные БД (PostgreSQL, MySQL 8+) поддерживают JSON-колонки. Это удобно: положил любой объект — и работает. Но это путь в никуда (подробно разберём в разделе 5.11).

5.5. NOT NULL vs NULL

NULL — это не "пусто" и не "ноль". NULL — это отсутствие значения, "неизвестно". Это принципиально важно.

NULL в вычислениях даёт NULL

Одно из самых неожиданных свойств NULL:

SELECT 100 + NULL;  -- Результат: NULL
SELECT NULL > 0;    -- Результат: NULL (не FALSE!)
SELECT NULL = NULL; -- Результат: NULL (не TRUE!)

Проверка WHERE price IS NULL — это не то же самое, что WHERE price = 0 или WHERE price = ''. Если в ваших данных есть NULL, все вычисления с ними дают NULL. Баланс клиента с NULL-полем — не 0, а "мы не знаем".

Разница между NULL, '' и 0

ЗначениеСмыслПример
NULLЗначение не установлено / неизвестноmiddle_name — у человека нет отчества или мы его не знаем?
'' (пустая строка)Значение установлено и оно пустоеnickname — пользователь сознательно очистил поле
0Числовое значение нольprice = 0 — товар бесплатный

Банковский пример: Поле discount_percent в кредитной заявке. NULL — скидка ещё не рассчитана. 0 — скидка рассчитана и равна нулю. 5 — скидка 5%. Если не различать NULL и 0, система может посчитать скидку как 0 для заявок, которые ещё не обработаны. Это потеря денег или ошибка.

Когда NULL допустим, когда нет

Общее правило: если у сущности атрибут обязательный — NOT NULL. Если опциональный — можно разрешить NULL.

CREATE TABLE users (
    id       BIGINT PRIMARY KEY,
    email    VARCHAR(254) NOT NULL,   -- обязателен
    phone    VARCHAR(20),             -- опционально, может быть NULL
    birthday DATE                     -- клиент может не указать
);

Senior: NULL — это всегда компромисс. С одной стороны, NULL — это честное "мы не знаем". С другой — NULL усложняет запросы (IS NULL, COALESCE, NULL-safe joins), замедляет индексацию (не все индексы работают с NULL) и порождает баги (NULL + число = NULL). В продакшене я предпочитаю NOT NULL везде, где возможно, а для опциональных полей ставлю значения по умолчанию: '', 0, false, 1970-01-01. Это не всегда корректно семантически, но спасает от скрытых багов.

5.6. UNIQUE — почему это важно

Если первичный ключ гарантирует уникальность одной колонки, то UNIQUE — это ограничение, которое защищает от дубликатов в любом наборе колонок.

Зачем: защита от дубликатов

Представьте регистрацию пользователей. Два человека одновременно заполняют форму с одинаковым email:

-- Без UNIQUE
CREATE TABLE users (id BIGINT PRIMARY KEY, email VARCHAR(254));
INSERT INTO users VALUES (1, 'ivan@example.com');
INSERT INTO users VALUES (2, 'ivan@example.com');  -- успех! Дубликат!

-- С UNIQUE
CREATE TABLE users (id BIGINT PRIMARY KEY, email VARCHAR(254) UNIQUE);
INSERT INTO users VALUES (1, 'ivan@example.com');
INSERT INTO users VALUES (2, 'ivan@example.com');  -- ОШИБКА! Дубликат отклонён

Ошибка: Нет UNIQUE на email. В системе появляются дубликаты. Начинается путаница: какой из двух Иванов заходил в систему? Кому отправлять письма? Приходится писать скрипты для чистки, merge пользователей — боль и риск.

Составной UNIQUE

Часто нужно гарантировать уникальность комбинации колонок:

-- Пользователь может добавить товар в корзину только один раз
CREATE TABLE cart_items (
    id         BIGINT PRIMARY KEY,
    user_id    BIGINT NOT NULL,
    product_id BIGINT NOT NULL,
    quantity   INTEGER NOT NULL DEFAULT 1,
    UNIQUE (user_id, product_id)
);

Разница: UNIQUE vs Primary Key

ХарактеристикаPRIMARY KEYUNIQUE
NULLНе допускает NULLДопускает NULL (в большинстве СУБД)
Количество на таблицуТолько одинМного
ИндексАвтоматически кластерный (часто)Автоматически некластерный
СмыслИдентификация строкиЗащита от дубликатов

Банковский пример: У одного клиента не может быть двух активных дебетовых карт с одинаковым номером. UNIQUE на card_number — защита от бага в процессинге, который создал бы две карты с одним номером. Это спасёт от двойного списания.

5.7. Индексы — ускоряем поиск

Что такое индекс простыми словами

Индекс — это как оглавление в книге. Если вам нужно найти главу про "индексы" в книге на 500 страниц, вы не будете листать каждую страницу. Вы откроете оглавление, найдёте номер страницы и перейдёте сразу. Индекс в БД работает точно так же.

Без индекса БД делает full table scan — читает каждую строку таблицы. С индексом — находит нужные страницы по оглавлению.

Без индекса: SELECT * FROM users WHERE email = 'ivan@example.com'; ┌──────┬─────────────────────┐ │ id │ email │ ├──────┼─────────────────────┤ │ 1 │ petr@example.com │ ← проверяем — не то │ 2 │ anna@example.com │ ← проверяем — не то │ 3 │ ivan@example.com │ ← нашли! │ ... │ ... │ │ 10^6 │ ... │ ← в худшем случае прочитаем 1 млн строк С индексом: Index (email): ┌─────────────────────┬──────┐ │ email │ стр. │ ├─────────────────────┼──────┤ │ anna@example.com │ 2 │ │ ivan@example.com │ 3 │ ← нашли за 2 шага (B-tree) │ petr@example.com │ 1 │ └─────────────────────┴──────┘

Как индекс ускоряет SELECT, замедляет INSERT/UPDATE

Ускорение SELECT: огромное. Вместо миллиона строк — десяток операций по B-tree.

Замедление INSERT/UPDATE/DELETE: при каждой вставке нужно обновить не только таблицу, но и все индексы на ней. Чем больше индексов — тем медленнее запись. На таблице с 10 индексами INSERT будет в 5-10 раз медленнее, чем на таблице без индексов.

Какие колонки индексировать

  • WHERE: колонки, которые часто фильтруют (WHERE status = 'active')
  • JOIN: колонки, по которым соединяют таблицы (внешние ключи)
  • ORDER BY: колонки, по которым сортируют (ORDER BY created_at DESC)
  • GROUP BY: колонки, по которым группируют

Ошибка: Индекс на каждую колонку. "Напишем индекс на всё, чтобы работало быстро". Результат: БД тратит ресурсы на поддержание кучи индексов, INSERT-ы падают по скорости, а SELECT-ы не всегда ускоряются (оптимизатор может выбрать не тот индекс). Индекс — это баланс, а не панацея.

Ошибка: Нет индекса на внешний ключ. Каждый JOIN по FK при отсутствии индекса — full table scan дочерней таблицы. На таблице с миллионом строк это секунды вместо миллисекунд.

Составные индексы: порядок колонок важен

-- Индекс на (status, created_at)
CREATE INDEX idx_orders_status_created ON orders (status, created_at);

-- Этот запрос использует индекс ЭФФЕКТИВНО:
SELECT * FROM orders
WHERE status = 'pending'
ORDER BY created_at DESC;

-- Этот запрос использует индекс НЕЭФФЕКТИВНО (status не указан):
SELECT * FROM orders
WHERE created_at > '2024-01-01';
-- Индекс бесполезен — первая колонка статуса не задана

Senior: Индекс — это всегда баланс скорости чтения и записи. На аналитических отчётах можно иметь много индексов (OLAP). На OLTP-системах с тысячами вставок в секунду каждый лишний индекс — потеря производительности. Смотрите на реальные запросы (slow query log), а не стройте индексы наугад.

5.8. Constraints — защита данных на уровне БД

Ограничения (constraints) — это правила, которые база данных принудительно применяет к данным. Это последняя линия обороны между вашими данными и хаосом.

CHECK — проверка на уровне строки

-- Цена товара должна быть положительной
CREATE TABLE products (
    id    BIGINT PRIMARY KEY,
    name  VARCHAR(200) NOT NULL,
    price DECIMAL(18,2) NOT NULL CHECK (price > 0)
);

-- Дата рождения не может быть в будущем
CREATE TABLE persons (
    id         BIGINT PRIMARY KEY,
    name       VARCHAR(200) NOT NULL,
    birth_date DATE NOT NULL CHECK (birth_date < CURRENT_DATE)
);

Банковский пример: CHECK (amount > 0) на таблице payments спасает от отрицательных платежей. Баг в коде может отправить транзакцию с amount = -1000. Если CHECK нет — деньги сняты. Если есть — ошибка и транзакция отклонена. CHECK стоит копейки, а спасает миллионы.

DEFAULT — значения по умолчанию

DEFAULT задаёт значение, которое будет вставлено, если колонка не указана:

CREATE TABLE orders (
    id         BIGINT PRIMARY KEY,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    status     VARCHAR(20) NOT NULL DEFAULT 'pending',
    total      DECIMAL(18,2) NOT NULL DEFAULT 0.00
);

Все ограничения — это constraints

На самом деле NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY — это все частные случаи constraints. БД не позволит нарушить ни одно из них.

Почему валидация в БД — последняя линия обороны

Валидация должна быть на всех уровнях:

  1. Фронтенд: мгновенная обратная связь пользователю
  2. Бэкенд (API): бизнес-логика и проверка прав доступа
  3. База данных: последняя защита от багов на верхних уровнях

Запомните: Любой программист может ошибиться. Фронтенд может не прислать валидацию. Бэкенд может пропустить некорректные данные. Constraint в БД — это гарантия, что "мусорные" данные туда не попадут никогда. Даже если весь код перепишут, constraint останется.

5.9. Soft Delete — почему записи не удаляют

Что такое soft delete

Вместо физического удаления строки (DELETE FROM) запись помечается как удалённая:

CREATE TABLE users (
    id         BIGINT PRIMARY KEY,
    name       VARCHAR(200) NOT NULL,
    email      VARCHAR(254) NOT NULL UNIQUE,
    deleted_at TIMESTAMP  -- NULL = активен, дата = удалён
);

-- "Удаление" пользователя:
UPDATE users SET deleted_at = NOW() WHERE id = 1;

-- Все запросы должны учитывать:
SELECT * FROM users WHERE deleted_at IS NULL;

Почему DELETE — риск

  • Каскадное удаление: удалили пользователя — потеряли его заказы, комментарии, историю операций
  • Потеря аудита: нельзя восстановить, кто что делал
  • Бизнес-требования: многие регуляторы требуют хранить данные N лет

Проблемы soft delete

  • UNIQUE conflicts: если email уникален, удалённый пользователь блокирует email для нового. Решение: составной UNIQUE (email, deleted_at) или NULL в unique constraint пропускается
  • Везде WHERE deleted_at IS NULL: забыл добавить в запрос — получил удалённые данные. Ошибка, которая может уйти в прод
  • Размер таблицы: удалённые записи не удаляются физически, таблица растёт

Альтернативы

СтратегияОписаниеКогда использовать
Hard DeleteФизическое удаление строкиЛоги, временные данные, сессии — то, что не жалко потерять; где аудит не нужен
Soft DeleteПометка deleted_atОсновные сущности (пользователи, заказы), когда данные должны быть доступны для восстановления
Archive TableПеред удалением копирование строки в архивную таблицу users_archiveКогда нужно хранить историю, но основная таблица должна быть чистой
Status = 'ARCHIVED'Вместо deleted_at используется колонка status с ENUMКогда удаление — это одно из состояний жизненного цикла сущности

Senior: Выбираем стратегию удаления в зависимости от требований. Для финансовой системы — только soft delete с полным аудитом. Для кэша или логов — hard delete с TTL (time-to-live). Для социальной сети — soft delete с возможностью восстановления (deleted_at = NULL). Универсального решения нет.

5.10. Аудит: кто, когда и что изменил

Любая вменяемая система должна знать, кто и когда создал запись и когда изменил её в последний раз. Это не "фича", а базовое требование для отладки, безопасности и регуляторов.

Минимальный аудит: created_at, updated_at

CREATE TABLE accounts (
    id         BIGINT PRIMARY KEY,
    balance    DECIMAL(18,2) NOT NULL,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

-- Триггер для автоматического обновления updated_at
CREATE OR REPLACE FUNCTION update_timestamp()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_accounts_updated
    BEFORE UPDATE ON accounts
    FOR EACH ROW EXECUTE FUNCTION update_timestamp();

Кто изменил: created_by, updated_by

Добавляем информацию о том, кто именно выполнил операцию:

CREATE TABLE documents (
    id         BIGINT PRIMARY KEY,
    title      VARCHAR(200) NOT NULL,
    content    TEXT,
    created_by BIGINT NOT NULL REFERENCES users(id),
    updated_by BIGINT REFERENCES users(id),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);

Продвинутый аудит: таблица audit_log

Просто хранить updated_at недостаточно — нужно видеть, что именно изменилось:

CREATE TABLE audit_log (
    id          BIGSERIAL PRIMARY KEY,
    table_name  VARCHAR(100) NOT NULL,
    record_id   BIGINT NOT NULL,
    operation   VARCHAR(10) NOT NULL CHECK (operation IN ('INSERT', 'UPDATE', 'DELETE')),
    old_data    JSONB,   -- предыдущие значения
    new_data    JSONB,   -- новые значения
    changed_by  BIGINT NOT NULL REFERENCES users(id),
    changed_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Пример записи:
-- table_name: 'accounts'
-- record_id: 42
-- operation: 'UPDATE'
-- old_data: {"balance": 1000.00}
-- new_data: {"balance": 950.00}
-- changed_by: 7 (пользователь)
-- changed_at: 2024-06-15 14:30:00+00

Банковский пример: Система процессинга платежей — аудит каждого изменения баланса обязателен. Регулятор (ЦБ) может запросить историю операций за 5 лет. Если у вас нет audit_log — вы не сможете предоставить отчёт. Штрафы могут достигать миллионов рублей. Аудит — это не опция, а необходимость.

Запомните: Проектируйте аудит с самого начала. Добавить audit_log в уже работающую систему — это миграция на миллион строк, изменение ORM, доработка API и тестов. Гораздо дешевле заложить created_at/updated_at в каждую таблицу сразу.

5.11. JSON в БД — удобно, но опасно

Почему разработчики используют JSON в БД

Причины простые и понятные:

  • Не нужно менять схему при добавлении нового поля — просто положил новый ключ в JSON
  • Удобно для хранения настроек, конфигов, метаданных
  • Быстрое прототипирование — не надо думать о нормализации
  • PostgreSQL JSONB поддерживает индексы (GIN) — можно искать внутри JSON

Проблемы JSON-колонок

ПроблемаОписание
Нет типовВ JSON можно положить и строку, и число, и null. Никто не проверит, что price — это число > 0. Ошибка в коде — и в базе лежит "price": "бесплатно".
Нет индексов (обычных)B-tree индекс на JSON не работает. GIN-индексы медленнее и не поддерживают все операции.
Нет JOINВы не можете сделать JOIN по полю внутри JSON. Только если не достанете его в виртуальную колонку.
Сложно мигрироватьКак переименовать поле внутри JSON? Только скриптом на миллион строк. Нет рефакторинга.
Нет документацииСхема JSON живёт только в коде. Новый разработчик не знает, какие ключи есть и что они значат.

Когда JSON оправдан

-- Хранение настроек — оправдано
CREATE TABLE user_settings (
    user_id BIGINT PRIMARY KEY REFERENCES users(id),
    settings JSONB NOT NULL DEFAULT '{}'
);
-- {
--   "theme": "dark",
--   "language": "ru",
--   "notifications": {"email": true, "sms": false}
-- }

-- Метаданные — оправдано
CREATE TABLE files (
    id          BIGINT PRIMARY KEY,
    filename    VARCHAR(255) NOT NULL,
    metadata    JSONB
);
-- {
--   "width": 1920,
--   "height": 1080,
--   "camera": "Canon EOS R5",
--   "gps": {"lat": 55.75, "lng": 37.61}
-- }

Антипаттерн: JSON вместо нормализации. Видел таблицу orders, где весь заказ хранился в одной JSON-колонке: товары, покупатель, адрес доставки, оплата. 100500 полей в одном JSON. Вы не можете сделать выборку "все заказы с товаром X" без полного сканирования. Не можете проверить referential integrity. Не можете наложить CHECK. Это свалка.

Senior: JSON в БД — это как свалка непонятно чего. Удобно скинуть всё в одну кучу, но когда нужно что-то найти — вы тоннами перебираете мусор. Если вы используете JSON, у вас должно быть железное обоснование: динамические атрибуты, настройки, метаданные из внешних систем. Если данные имеют структуру — создайте колонки. Пожалейте будущего разработчика (и себя через полгода).

Итоги главы

Проектирование базы данных — это не про SQL и не про выбор СУБД. Это про мышление структурами. Каждый раз, когда вы добавляете колонку, вы принимаете решение, которое будет влиять на систему годами.

Чек-лист проектирования таблицы

  • Определена сущность (что именно хранит таблица?)
  • Есть Primary Key (суррогатный, BIGINT или UUIDv7)
  • NOT NULL на обязательных полях
  • UNIQUE на бизнес-уникальных полях (email, номер карты)
  • FOREIGN KEY на ссылочные колонки
  • CHECK на диапазоны и условия (price > 0)
  • DEFAULT для полей со значением по умолчанию
  • Типы данных обоснованы (не DECIMAL для денег → ошибка)
  • Индексы на WHERE, JOIN, ORDER BY колонки
  • created_at / updated_at — минимальный аудит
  • Стратегия удаления выбрана (soft delete или archive)
  • JSON только для настроек/метаданных, не для бизнес-данных

Что должен понять аналитик

Главный вывод: База данных — не просто место для хранения. Это слой приложения, который обеспечивает контракты между данными. Если аналитик предлагает "просто добавить поле в JSON" или "у нас не будет дубликатов, код проверит" — он создаёт технический долг. Понимание схемы БД — это понимание того, как устроен бизнес: какие сущности есть, как они связаны, какие правила работают на уровне данных. Аналитик, который мыслит схемами, — ценный член команды.
