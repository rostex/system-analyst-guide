Техническое мышление аналитика

Практическое руководство для Business Analyst и System Analyst

Как мыслить как инженер · Понимать почему, а не только что · Проектировать требования без замечаний

🎯 Цель книги: Сформировать техническое мышление аналитика. После прочтения вы будете понимать не только что нужно делать, но и почему именно так принято в современной разработке. Вы начнёте задавать те же вопросы, что задаёт Senior-разработчик перед началом реализации.

📚 О книге

Эта книга — не справочник по REST API и не учебник по SQL. Это практическое руководство, которое объясняет:

  • как думает разработчик и архитектор;
  • почему возникают архитектурные ограничения;
  • почему разработчики задают аналитикам десятки вопросов;
  • как проектировать требования так, чтобы разработчики принимали их без замечаний.

Главный принцип книги — каждая тема отвечает на вопрос ПОЧЕМУ, а не только ЧТО.

👥 Для кого

Business Analyst · System Analyst · Functional Analyst

Также будет полезна: Product Owner, Project Manager, Junior Backend Developer, QA Engineer

📖 Как читать

Последовательно, глава за главой. Каждая глава — логически завершённый материал с примерами, ошибками, решениями и чек-листами.

📑 Содержание

01
Как думает разработчик
Внутренняя кухня разработчика. Почему "просто добавить поле" может занять 3 дня. Как разработчик оценивает сроки. Что такое технический долг.
02
REST API без магии
Как устроен REST API. HTTP-методы, статус-коды, URL-дизайн. Идемпотентность, пагинация, фильтрация. Как должен выглядеть контракт API в документации.
03
JSON как контракт
Типы данных в JSON. Nullable vs Optional. Enum, UUID, ISO 8601. Decimal, Float, Double — почему одни теряют деньги, а другие нет. OpenAPI и версионирование.
04
Что происходит внутри Backend
Путь запроса от клиента до БД. Controller, DTO, Service, Repository. Транзакции и ACID. Обработка ошибок. Почему валидация нужна на каждом уровне.
05
Frontend глазами аналитика
Как работает браузер, API между фронтом и бэком, состояния UI, UX как часть требований.
06
Проектирование базы данных
Первичные и внешние ключи. Типы данных, NOT NULL vs NULL, UNIQUE. Индексы, Constraints. Soft Delete, аудит. JSON в БД — удобно, но опасно.
07
Нормализация данных
Три нормальные формы на реальных примерах. Когда нормализация — зло. Денормализация для производительности. Баланс между консистентностью и скоростью.
08
Производительность
N+1 проблема, индексы, JOIN, пагинация. Кэширование. Batch Processing. Как требования аналитика влияют на производительность системы.
09
Микросервисы
Монолит vs микросервисы. Kafka и очереди сообщений. Event Driven архитектура. Владельцы данных. Saga pattern. Когда микросервисы — плохое решение.
10
Интеграции
Синхронные и асинхронные интеграции. REST vs gRPC vs GraphQL. Обработка ошибок, таймауты, retry. Eventual consistency. Идемпотентность.
11
Как писать хорошие требования
Business vs Technical requirements. Структура хорошего требования. Явные и неявные требования. User Stories. Граничные случаи. Чек-лист проверки.
12
Антипаттерны аналитиков
10 самых частых ошибок: "это же просто добавить поле", "сделайте как в старом интерфейсе", "это временное решение" и другие. Разбор каждой с последствиями.
13
Чек-листы Senior аналитика
Готовые инструменты для ежедневной работы. Чек-листы проверки требований, API-контрактов, интеграций. Roadmap от Junior до Senior аналитика.

✅ Главная идея книги: Аналитик с инженерным мышлением — самый ценный член команды. После каждой главы вы будете смотреть на систему глазами разработчика и задавать правильные вопросы до того, как начнётся реализация.
