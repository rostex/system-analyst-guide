Глава 4: Что происходит внутри Backend

Путешествие запроса от клиента до базы данных и обратно

Аналог Backend — это ресторан. Клиент (frontend) делает заказ (запрос) официанту (API Gateway / Controller), который передаёт его на кухню (Service Layer). Там повара (Business Logic) готовят блюдо, используя продукты (данные) из холодильника (Database). Когда блюдо готово, официант несёт его обратно клиенту.

Каждый день аналитик описывает требования к бэкенду: «сервер должен сохранить заказ», «проверить баланс пользователя», «отправить уведомление». Но что на самом деле происходит внутри этого чёрного ящика? Почему разработчики не могут «просто взять и сохранить»? Зачем столько слоёв, классов, транзакций?

В этой главе мы заглянем под капот backend-разработки и проследим путь одного запроса от клиента до базы данных и обратно. Вы поймёте, какие слои существуют, зачем каждый из них нужен и как это знание поможет вам писать более точные требования.

4.1. Путь запроса: от клиента до базы и обратно

Любой запрос к бэкенду проходит несколько стадий обработки. Даже самая простая операция — «получить профиль пользователя» — проходит через цепочку слоёв. Вот как это выглядит:

Клиент (Frontend / Mobile / Postman) │ │ HTTP-запрос (GET /api/users/42) ▼ ┌──────────────────────┐ │ Controller │ ← принимает запрос, парсит JSON, проверяет токен └────────┬─────────────┘ │ DTO ▼ ┌──────────────────────┐ │ Service │ ← бизнес-логика: проверка прав, расчёты, правила └────────┬─────────────┘ │ Entity / Model ▼ ┌──────────────────────┐ │ Repository │ ← работа с БД: CRUD-операции └────────┬─────────────┘ │ SQL ▼ ┌──────────────────────┐ │ Database (SQL) │ ← хранилище данных (PostgreSQL, MySQL, ...) └──────────────────────┘ │ ▼ Данные возвращаются обратно по цепочке Клиент получает JSON-ответ

Каждый слой решает свою узкую задачу. Давайте посмотрим на таблицу ответственности:

Слой Задача Что делает
Controller Входные ворота Принимает HTTP-запрос, парсит JSON, проверяет авторизацию, вызывает Service
DTO Объект передачи Определяет формат данных для API — что отправить и что получить
Service Бизнес-логика Расчёты, проверки, оркестрация нескольких операций
Repository Доступ к данным CRUD, SQL-запросы, скрытие деталей хранения
Entity Модель данных Отображение таблицы БД на объект в коде
Database Хранилище Хранение, индексация, целостность, транзакции

Почему так много слоёв? Каждый слой — это точка изменения. Если завтра вы поменяете базу данных с PostgreSQL на MongoDB, изменится только Repository. Controller и Service останутся нетронутыми. Если изменится бизнес-правило — вы правите только Service. Разделение слоёв — это не «усложнение ради усложнения», это управление сложностью.

Теперь пройдёмся по каждому слою детально.

4.2. Controller — входные ворота

Controller (контроллер) — это первый слой, который встречает запрос. Его задача — принять HTTP-запрос от клиента, быстро проверить его «на входе» и передать дальше.

Что происходит в Controller

  1. Парсинг входящих данных — преобразование JSON-тела запроса в DTO.
  2. Базовая валидация — проверка обязательных полей, формат email, длина строки.
  3. Авторизация — проверка JWT-токена, прав доступа.
  4. Вызов Service — передача управления слою бизнес-логики.
  5. Формирование ответа — упаковка результата в HTTP-ответ с нужным статус-кодом.

❌ Ошибка: контроллер, который считает скидку

Встречается сплошь и рядом. Контроллер сам вычисляет скидку для пользователя, проверяет баланс и решает, можно ли оформить заказ. Почему это плохо? Бизнес-логика оказывается «зашита» в слой, который должен заниматься только маршрутизацией. Если завтра изменится правило расчёта скидки — придётся искать это в контроллерах, а не в сервисе.

Правильный контроллер — это тонкий слой. Он похож на пульта охраны в здании: проверил пропуск, открыл дверь, указал, в какой кабинет идти. И всё. Внутренние дела здания пульт не решает.

Пример: правильный контроллер

// Контроллер НЕ должен содержать бизнес-логику
@PostMapping("/orders")
public ResponseEntity<OrderResponse> createOrder(@RequestBody @Valid CreateOrderRequest request) {
    // 1. Парсинг и валидация — уже сделаны (@Valid превращает JSON в DTO)
    // 2. Вызов сервиса — вся логика там
    OrderResponse result = orderService.createOrder(request);
    // 3. Ответ
    return ResponseEntity.status(201).body(result);
}

🧠 Как думает Senior Controller — это только маршрутизация. Если вы видите в контроллере if, for или switch с бизнес-условием — вы видите проблему. Контроллер должен умещаться в 3–5 строк без логики.

4.3. DTO — объект передачи данных

DTO (Data Transfer Object) — это объект, который определяет, какие данные попадают в API-запрос и возвращаются в ответе. Это «форма для заполнения».

Зачем нужен DTO

Самая частая ошибка новичков — отдавать клиенту entity (объект базы данных) напрямую. Чем это опасно?

Почему нельзя отдавать entity из БД напрямую

  • Безопасность: в entity может быть password_hash, внутренний id, служебные флаги. Вы случайно отправите их клиенту.
  • Избыточность: у User в БД 20 полей, а на фронте нужны только 5. Зачем гонять лишние мегабайты?
  • Зависимость: если вы измените структуру БД, изменится API. А это две разные сущности.

Схема преобразования

Entity (БД) DTO (API) JSON (клиент) ┌────────────┐ ┌────────────┐ ┌────────────────┐ │ id │ │ id │ │ { │ │ email │ ──▶ │ email │ ──▶ │ "id": 42, │ │ name │ │ name │ │ "email":... │ │ password │ └────────────┘ │ "name":... │ │ created_at │ │ } │ │ deleted │ └────────────────┘ └────────────┘ ╰── password_hash НЕ попадает в DTO — это внутреннее поле

🏦 Пример: банк Представьте, что банковское API возвращает entity Card напрямую. В ответе окажется полный номер карты, CVV-код и срок действия. Вместо этого API возвращает DTO с маскированным номером: "cardNumber": "**** **** **** 1234". Внутренние поля (PIN, CVV, полный номер) никогда не покидают сервер.

⚠️ Security: никогда не отдавать внутренние поля Явно опишите в требованиях, какие поля не должны попадать в API-ответ. Пароли, хеши, токены, внутренние идентификаторы, служебные флаги (is_deleted, internal_note) — всё это должно остаться внутри бэкенда. Если аналитик не пропишет это в требованиях — разработчик может «забыть» сделать DTO.

Аналогия DTO — это форма для визы. Консульство не просит вас показать всю вашу медицинскую карту. Только страницу с паспортными данными и фото. Так и DTO отдаёт только то, что нужно для конкретной операции.

4.4. Service — мозг приложения

Service (сервис) — это слой, где живёт бизнес-логика. Именно здесь принимаются решения: можно ли оформить заказ, как рассчитать скидку, достаточно ли денег на счету.

Факт В хорошо спроектированном приложении 70–80% всех изменений происходит именно в Service-слое. Контроллеры и репозитории меняются редко.

Что делает Service

  • Проверка бизнес-правил: «баланс не может быть отрицательным», «заказ можно отменить только в течение часа».
  • Расчёты: налоги, скидки, комиссии.
  • Оркестрация: вызвать несколько репозиториев, объединить данные, выполнить действия в правильном порядке.
  • Транзакции: обеспечить, чтобы несколько операций выполнились атомарно.

Пример: сервис перевода денег

public class PaymentService {

    @Transactional
    public TransferResult transferMoney(TransferRequest request) {
        // 1. Проверка бизнес-правил
        Account from = accountRepository.findById(request.getFromId());
        Account to = accountRepository.findById(request.getToId());

        if (from.getBalance() < request.getAmount()) {
            throw new InsufficientFundsException("Недостаточно средств");
        }

        // 2. Выполнение операций
        from.withdraw(request.getAmount());
        to.deposit(request.getAmount());

        // 3. Сохранение
        accountRepository.save(from);
        accountRepository.save(to);

        // 4. Аудит
        auditLogger.log("Перевод", from.getId(), to.getId(), request.getAmount());

        return new TransferResult("Успешно");
    }
}

🏦 Пример: перевод денег Представьте, что при переводе 1000 рублей мы списали деньги с карты А, но сервер упал до того, как зачислил их на карту Б. Деньги исчезли. Транзакция гарантирует: или произойдут оба действия, или ни одного. В программировании это называется атомарность.

❌ Ошибка: бизнес-логика в SQL-запросе Иногда разработчики пишут сложные вычисления прямо в SQL: UPDATE users SET bonus = salary * 0.1 + (SELECT ...). Это ошибка: бизнес-логика оказывается «зашита» в запрос, её нельзя покрыть юнит-тестами, её сложно менять. Любое изменение правила требует правки SQL, а не кода. Бизнес-логика должна быть в Service, а SQL — только операции чтения/записи.

🧠 Как думает Senior Один метод сервиса — одна бизнес-операция. Нельзя делать гигантский метод на 500 строк. Если операция сложная — разбейте на несколько приватных методов. Каждый метод должен отвечать на вопрос «что произошло?» на уровне бизнеса, а не на уровне кода.

4.5. Repository — доступ к данным

Repository (репозиторий) — это слой, который абстрагирует работу с базой данных. Он скрывает от сервиса, какая именно БД используется: PostgreSQL, MySQL, MongoDB или даже обычный файл.

CRUD — основа работы с данными

Операция SQL HTTP-метод Пример из жизни
Create INSERT POST Создать заказ
Read SELECT GET Получить профиль
Update UPDATE PUT / PATCH Изменить адрес доставки
Delete DELETE DELETE Удалить товар из корзины

Зачем нужен отдельный слой для данных

Без Repository бизнес-логика и SQL смешиваются. Service начинает содержать запросы — и через полгода вы не можете поменять БД, потому что все запросы размазаны по коду. Repository — это контракт: «я умею сохранять пользователя, искать по id, удалять. Как именно — неважно».

SQL injection — что это и как защититься

SQL injection — это когда пользователь вводит в поле «имя» не имя, а SQL-команду: '; DROP TABLE users; --. Если бэкенд просто подставляет значение в строку запроса, база данных выполнит эту команду. Защита: параметризованные запросы (prepared statements). Современные ORM делают это автоматически, но аналитик должен знать, что передача сырых данных в SQL — это дыра в безопасности.

🧠 Как думает Senior Repository — это контракт с данными. Сервис говорит: «дай мне пользователя с id = 42». Repository решает: это будет SQL-запрос к PostgreSQL, или чтение из кэша Redis, или вызов внешнего API. Сервису всё равно. Контракт важен, реализация — деталь.

4.6. Модели и Entity

Entity (сущность) — это объект, который напрямую отражает таблицу в базе данных. Каждая строка таблицы — один экземпляр entity.

Пример: User в БД

Таблица users:

Поле Тип Назначение
id BIGINT Внутренний идентификатор
email VARCHAR(255) Email пользователя
name VARCHAR(100) Имя
password_hash VARCHAR(255) Хеш пароля (никому не отдаём!)
phone VARCHAR(20) Телефон
birth_date DATE Дата рождения
is_active BOOLEAN Флаг активности
created_at TIMESTAMP Дата регистрации

В entity будет ровно те же поля, что в таблице. Но API (через DTO) отдаёт только 5 из них:

{
  "id": 42,
  "email": "user@example.com",
  "name": "Иван Петров",
  "phone": "+7-999-123-45-67",
  "isActive": true
}

Что такое ORM ORM (Object-Relational Mapping) — это библиотека, которая автоматически преобразует строки из БД в объекты в коде. Вы пишете userRepository.findById(42) — ORM выполняет SELECT * FROM users WHERE id = 42 и возвращает объект User. Разработчику не нужно писать SQL вручную. Примеры: Hibernate (Java), Entity Framework (.NET), SQLAlchemy (Python), TypeORM (TypeScript).

Почему entity не должен быть DTO и наоборот У них разные цели. Entity отражает структуру БД. DTO — структуру API. Если вы используете entity как DTO, вы «светите» внутренние поля и привязываете API к схеме БД. Если вы используете DTO как entity — вы теряете данные из БД, которые могут понадобиться внутри.

4.7. Валидация — где и что проверяется

Валидация данных происходит на трёх уровнях. Аналитик должен понимать, на каком уровне что проверяется, и явно указывать это в требованиях.

Три уровня валидации

Уровень Что проверяет Пример
DTO (Controller) Формат данных, синтаксис Email содержит @, поле обязательно, длина не больше 255
Service Бизнес-правила Баланс > 0, статус заказа позволяет отмену, товар есть на складе
Database Constraints, целостность UNIQUE(email), NOT NULL, FOREIGN KEY, CHECK (price > 0)

❌ Ошибка: полагаться только на frontend-валидацию Фронтенд можно взломать, отключить JavaScript или отправить запрос напрямую через Postman/cURL. Валидация на клиенте — для удобства пользователя. Валидация на бэкенде — для безопасности системы. Никогда не считайте, что «фронт уже проверил».

🏦 Пример: negative price Представьте, что пользователь указывает цену товара -1000 рублей.

  • DTO: проверит, что цена — это число. Но число может быть отрицательным.
  • Service: проверит, что цена > 0. Это бизнес-правило.
  • БД: constraint CHECK (price > 0) — страховка на случай, если ошибка проскочила через код.

Только все три уровня вместе дают надёжную защиту.

4.8. Обработка ошибок

Ошибки неизбежны. Пользователь указал неверный пароль, сервер перегружен, товара нет на складе. Архитектура бэкенда должна обрабатывать ошибки предсказуемо и безопасно.

Как ошибка проходит слои

DB ──▶ Repository ──▶ Service ──▶ Controller ──▶ Клиент Пример: 1. DB: нарушен CONSTRAINT UNIQUE — дубликат email 2. Repository: ловит SQLException, оборачивает в DataAccessException 3. Service: ловит DataAccessException, формирует бизнес-ошибку "Email уже занят" 4. Controller: ловит исключение, возвращает HTTP 409 Conflict 5. Клиент: получает JSON { "error": "Email уже занят", "code": "EMAIL_DUPLICATE" }

Маппинг ошибок в HTTP-статусы

HTTP-статус Когда используется Пример
200 OK Всё хорошо Успешный GET-запрос
201 Created Сущность создана Оформлен заказ
400 Bad Request Неверный формат запроса Некорректный JSON, пропущено поле
401 Unauthorized Не аутентифицирован Токен истёк или отсутствует
403 Forbidden Нет прав Пользователь не админ, но пытается удалить юзера
404 Not Found Ресурс не найден Нет заказа с таким id
409 Conflict Конфликт данных Дубликат email, попытка отменить уже отправленный заказ
422 Unprocessable Entity Бизнес-правило нарушено Недостаточно средств
500 Internal Server Error Неизвестная ошибка сервера Упала БД, NullPointerException

Почему стектрейс нельзя отдавать клиенту Stack trace — это внутренняя кухня: имена классов, версии библиотек, пути на сервере. Злоумышленник может использовать эту информацию для атаки. Клиент должен видеть только: «что пошло не так» и «код ошибки». Подробности — в логи сервера.

Аналогия Сообщение об ошибке в самолёте для пассажиров: «Уважаемые пассажиры, отказал двигатель №2, мы возвращаемся в аэропорт». А не «java.lang.NullPointerException в модуле FuelInjectorController при вызове метода getEngineStatus()». Клиенту нужен понятный ответ, а не техническая диагностика.

4.9. Транзакции и ACID

Транзакция — это группа операций, которые выполняются как одно целое. Если хотя бы одна операция не удалась — отменяются все.

Аббревиатура ACID описывает свойства, которым должна удовлетворять транзакция:

Свойство Что означает Пример
Atomicity
Атомарность		 Всё или ничего. Операция либо выполняется полностью, либо не выполняется вовсе. Перевод денег: списание + зачисление. Если упало — деньги возвращаются.
Consistency
Согласованность		 Данные всегда корректны. После транзакции все constraints соблюдены. Сумма балансов до перевода = сумме после. Деньги не появляются из ниоткуда.
Isolation
Изоляция		 Параллельные транзакции не мешают друг другу. Два человека одновременно покупают последний билет — купит только один.
Durability
Долговечность		 Если транзакция подтверждена — данные не пропадут даже при отключении питания. После ответа «оплата прошла» деньги точно зачислены, даже если сервер тут же выключится.

🏦 Пример: перевод с карты на карту

  1. Вы нажимаете «Перевести 5000 рублей другу».
  2. Бэкенд открывает транзакцию.
  3. Списывает 5000 с вашей карты.
  4. Зачисляет 5000 на карту друга.
  5. Если всё ок — транзакция подтверждается (commit).
  6. Если на шаге 4 ошибка (карта друга заблокирована) — транзакция откатывается (rollback), и ваши 5000 возвращаются.

Без транзакции вы бы потеряли деньги: с вашей карты списалось, а другу не пришло.

Почему аналитик должен понимать ACID Когда вы пишете требование «при переводе денег нужно списать с одной карты и зачислить на другую» — вы неявно описываете транзакцию с атомарностью. Если вы этого не пропишете, разработчик может не реализовать транзакцию, и вы получите баг на продакшене. Указывайте: «операция должна быть атомарной: или выполняются оба действия, или ни одного». Это заставит разработчика использовать транзакцию.

Итоги главы

Чек-лист: что проверить в требованиях для backend

  • DTO: какие поля отдаём клиенту, какие НЕ отдаём (security)?
  • Валидация: прописана валидация на всех трёх уровнях (формат, бизнес-правила, БД)?
  • Транзакции: нужно ли объединять несколько операций в одну транзакцию?
  • Обработка ошибок: какой HTTP-статус и какое сообщение получает клиент в каждом сценарии?
  • Бизнес-логика в SQL: нет ли расчётов, которые лучше вынести в Service?
  • SQL injection: предусмотрена ли защита (параметризованные запросы)?
  • Entity vs DTO: не светим ли внутренние поля БД через API?
  • Изоляция: что будет, если два пользователя одновременно выполнят одно и то же действие?

Что должен понять аналитик

  1. Слои — это не усложнение, а защита. Каждый слой решает свою задачу. Чем чётче разделение, тем проще менять систему.
  2. Бизнес-логика живёт в Service. Не в SQL, не в контроллере, не на фронтенде.
  3. DTO защищает данные. Никогда не отдавайте клиенту то, что ему не нужно.
  4. Транзакции спасают деньги. Любая операция с деньгами, статусами или важными данными должна быть атомарной.
  5. Валидация на всех уровнях. Фронтенд — для удобства, бэкенд — для безопасности, БД — для гарантии.
  6. Ошибки должны быть человекочитаемыми. Клиент получает понятный ответ, разработчик — логи.

🧠 Резюме для Senior Analyst Понимание внутреннего устройства backend превращает аналитика из «передатчика пожеланий» в полноценного участника архитектурных обсуждений. Когда вы говорите «это должно быть в DTO», «здесь нужна транзакция» или «эта валидация — бизнес-правило, она должна быть в Service» — разработчики воспринимают вас как равного. Не бойтесь технических деталей: за ними стоит логика, а логика — это ваш язык.
